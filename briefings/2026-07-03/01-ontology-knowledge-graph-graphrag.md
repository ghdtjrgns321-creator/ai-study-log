# 오늘의 AI 개념: 온톨로지 · 지식그래프 · GraphRAG

> 작성일: 2026-07-03 (2026-07-24 심화 개편) · 분류: ai-concept

## 한 줄 정의

데이터를 표가 아니라 개체와 개체 사이의 관계망으로 저장하고, 그 관계망을 검색해 AI 답변의 근거로 삼는 방식이다.

## 쉬운 설명

지식그래프(Knowledge Graph)는 정보를 "개체 — 관계 — 개체" 형태로 연결해 저장하는 구조다. 예를 들어 "삼성전자 — 공급받는다 → TSMC", "K-IFRS 1115호 — 적용된다 → 수익인식"처럼 점(개체)과 선(관계)으로 세상을 그린다.

온톨로지(Ontology)는 그 그래프를 그리기 위한 설계도다. "회사라는 개체는 어떤 속성을 가지며 어떤 관계를 맺을 수 있는가"를 미리 규정한 스키마에 해당한다. 관계형 데이터베이스로 비유하면 온톨로지는 테이블 설계(스키마)이고, 지식그래프는 그 설계에 따라 실제로 채워진 데이터다.

GraphRAG는 이 지식그래프를 RAG(Retrieval-Augmented Generation, 검색해 온 자료를 근거로 답변을 생성하는 기법)에 결합한 것이다. 일반 RAG가 문서를 잘게 쪼갠 조각을 벡터 유사도로 검색한다면, GraphRAG는 개체와 관계, 경로까지 함께 검색한다. 그 덕분에 "A와 B를 거쳐 C에 도달하는" 다단계 추론 질문에서 더 정확하고 근거를 추적하기 쉬운 답을 낸다.

회계·감사 영역에서 특히 주목받는 이유는 거래처·계열사·전표 사이의 관계 자체가 분석 대상이기 때문이다.

## 동작 원리

Microsoft GraphRAG 기준으로, 비용 큰 작업을 질문 전(인덱싱)에 미리 해 두고, 질문 유형에 따라 그래프를 다르게 탐색하는 2단 구조다.

```
[인덱싱 — 질문이 오기 전에 미리 수행]
원문 문서 ─▶ ①TextUnit 분할 ─▶ ②LLM이 개체·관계·주장 추출 ─▶ ③지식그래프 구성
                                                                  │
                              ④Leiden 알고리즘으로 커뮤니티(주제 군집) 계층 탐지
                                                                  │
                              ⑤커뮤니티마다 LLM 요약을 미리 생성(하위→상위 순)

[쿼리 — 질문 유형에 따라 경로 선택]
질문 ─▶ 데이터 전체 조망형? ─┬─예──▶ Global Search: 커뮤니티 요약들을 취합해 답
                             └─아니오▶ Local Search: 해당 개체로 진입해
                                       이웃 개체·관계로 확장하며 근거 수집
```

1. **인덱싱** — 문서를 TextUnit이라는 분석 단위로 쪼개고, LLM이 각 단위에서 개체(entity)·관계(relationship)·주장(claim)을 추출해 그래프를 만든다. 개체마다 출처 TextUnit이 기록돼 답변 근거를 원문까지 역추적할 수 있다.
2. **커뮤니티 탐지·요약** — Leiden 군집화 알고리즘으로 밀접한 개체들을 커뮤니티로 묶고, 커뮤니티마다 요약문을 미리 만들어 둔다. 이 사전 요약이 "전체 주요 주제 5개는?" 같은 조망형 질문을 가능하게 한다.
3. **쿼리** — 조망형 질문은 커뮤니티 요약을 모아 답하는 Global Search, 특정 개체 질문은 이웃으로 확장하는 Local Search를 쓴다. 둘을 절충한 DRIFT Search, 단순 벡터 검색인 Basic Search도 있다.

요컨대 일반 RAG가 "질문과 비슷한 조각 찾기"라면, GraphRAG는 "미리 그려 둔 관계 지도 위를 걸어다니며 근거 모으기"다.

## 구체 예시·사례

Microsoft Research의 공개 실험을 따라가 보자. 2023년 6월 러시아·우크라이나 뉴스 기사 수천 건(VIINA 데이터셋)을 넣고 "Novorossiya(노보로시야)가 무엇을 했는가?"라고 물었다.

- **일반 RAG**: 이 단어와 활동 내용을 함께 담은 문서 조각이 없어 답을 내지 못했다. 벡터 검색은 질문과 "비슷한 조각"만 가져오기 때문이다.
- **GraphRAG**: 인덱싱 때 이미 'Novorossiya' 개체 노드가 만들어져 있고, 여러 기사에 흩어진 활동들이 관계 간선으로 연결돼 있다. 이 노드에서 간선을 따라가며 근거를 모아, 원문 출처 링크가 달린 답을 만들었다.

감사 업무로 치환하면 "이 거래처와 특수관계에 있는 회사들을 거친 거래 흐름은?" 같은 질문과 구조가 같다. 어느 한 전표에도 전체 그림이 없고, 흩어진 관계를 이어 붙여야 답이 나오는 유형이다.

## 비슷한 것과 비교

| 축          | 일반(벡터) RAG                     | GraphRAG — Local                  | GraphRAG — Global                  |
| ----------- | ---------------------------------- | --------------------------------- | ---------------------------------- |
| 검색 단위   | 문서 조각(청크)의 벡터 유사도      | 개체 노드 + 이웃 관계             | 사전 생성된 커뮤니티 요약          |
| 잘하는 질문 | 답이 한두 조각 안에 있는 사실 질문 | 특정 개체 중심의 다단계 관계 질문 | "전체 주제·경향은?" 같은 조망 질문 |
| 한계        | 흩어진 정보 연결·전체 요약에 약함  | 그래프 품질(추출 정확도)에 의존   | 인덱싱 비용 큼, 세부 사실엔 과함   |
| 언제 쓰나   | FAQ·단일 문서 조회                 | 관계 추적(인물·거래처·계정)       | 데이터셋 전반 분석·보고            |

선택 기준: 답이 한 조각 안에 있으면 벡터 RAG로 충분하고, 흩어진 관계를 잇거나 전체를 조망해야 하면 GraphRAG를 쓴다. 실무에서는 둘을 섞어 질문 유형별로 라우팅하는 구성이 흔하다.

## 왜 지금 중요한가

Microsoft Research는 일반 RAG의 두 가지 구조적 한계 — ①흩어진 정보를 연결하는 질문에 약하고 ②데이터셋 전체를 요약하는 질문에 실패한다 — 를 지적하며 GraphRAG를 제안했고, 오픈소스로 공개해 파이프라인을 문서화했다. 사내 문서에 대해 "연결"과 "조망"이 필요한 기업 환경이 정확히 이 한계에 걸리기 때문에, RAG 고도화 논의의 중심축이 됐다.

다만 확산의 제약도 분명하다. SK 데보션 기술 블로그는 지식그래프 구축 자체의 난이도와 추론 연산 비용 때문에 아직 광범위한 도입은 제한적이라고 짚는다.

## 회계법인 AI 직무 연결 포인트

특수관계자 거래 탐지가 가장 직접적인 접점이다. 회사·거래처·계열사·임원을 그래프로 연결하면, 표로는 드러나지 않는 순환·우회 거래 경로를 멀티홉 추론으로 찾아낼 수 있다. "이 거래처의 대표가 감사 대상 회사 임원과 겹치는가"는 본질적으로 그래프 경로 질문이다.

전표 흐름 추적에도 같은 구조가 쓰인다. 계정·전표·증빙을 개체로 연결하면 입력부터 재무제표 반영까지의 경로를 따라가며 이상 지점을 짚을 수 있고, GraphRAG의 출처 역추적은 감사 증거 문서화 요구와 잘 맞는다.

감사 문서 질의도 유력하다. 계약서·공시·내부규정을 지식그래프로 묶으면 "이 거래에 적용되는 기준서와 근거 문서는 무엇인가" 같은 다단계 질문에 근거를 붙여 답할 수 있다. 온톨로지 설계 경험은 감사 도메인의 데이터 모델링(계정 체계·관계 정의) 역량으로 직결된다.

## 핵심 용어·논쟁

- **트리플(Triple)** — "개체–관계–개체" 한 쌍. 지식그래프의 최소 저장 단위.
- **TextUnit** — GraphRAG 인덱싱의 기본 청크이자 근거 역추적 단위.
- **커뮤니티 탐지(Community Detection)** — 밀접한 개체들을 군집으로 묶는 것. GraphRAG는 Leiden 알고리즘을 쓴다.
- **멀티홉 추론(Multi-hop Reasoning)** — 여러 관계 간선을 건너뛰며 답을 조립하는 추론.
- **Global / Local Search** — 커뮤니티 요약 기반 조망형 검색 / 개체 중심 확장형 검색.

논쟁 지점은 비용 대비 효과다. 인덱싱 때 코퍼스 전체를 LLM으로 처리해야 하고 그래프 갱신·추론 비용도 커서, "어떤 질문 유형에 벡터 RAG 대비 투자 가치가 있는가"가 도입 판단의 핵심이다. Microsoft도 평가 프레임워크를 개발 중이라고 밝혀 표준화된 정량 평가는 진행형이다.

## 자료 깊이 읽기

### GraphRAG: Unlocking LLM discovery on narrative private data (Microsoft Research 블로그) — 영어 / 텍스트 / 중급
GraphRAG 원 제안 글. 일반 RAG가 실패하는 두 유형(흩어진 정보 연결, 전체 요약)을 Novorossiya 실제 질의로 보여주고, 개체·관계 추출 → 상향식 군집화 → 커뮤니티별 사전 요약이라는 3단계 구성을 설명한다. 포괄성·근거 제시·다양성 지표에서 일반 RAG를 일관되게 앞섰다고 보고하되(자체 평가), 평가 프레임워크는 아직 개발 중임을 인정한다.

### GraphRAG 공식 문서 Overview (microsoft.github.io/graphrag) — 영어 / 공식 문서 / 중급
오픈소스 GraphRAG의 파이프라인 문서. 인덱싱은 TextUnit 분할 → 개체·관계·주장 추출 → Leiden 기반 계층 커뮤니티 탐지 → 상향식 커뮤니티 요약 순서로 진행되고, 쿼리 모드는 Global·Local·DRIFT·Basic 4가지를 제공한다. 벡터 유사도에만 의존하는 기본 RAG의 한계를 구조화된 계층으로 보완한다는 설계 의도가 명시돼 있다.

### 지식그래프·GraphRAG: 구조화된 지식의 발견 (SK 데보션) — 한국어 / 텍스트 / 입문
지식그래프를 "별자리" 비유로 설명한다 — 낱개 별(데이터)은 연결돼 별자리(관계 구조)가 될 때 의미를 갖는다. 일반 RAG가 두 영역을 동시에 요구하는 질문(식품 처리 절차 + 환불 규정)에 약하다는 예시를 들고, 해법으로 멀티홉 추론과 계층 군집화 기반 전역 검색을 소개한다. 그래프 구축 난이도와 추론 비용 때문에 확산이 제한적이라는 현실적 한계도 짚는다.

## 자가 점검 질문

1. 일반 RAG와 GraphRAG의 차이를 면접관에게 1분 안에 설명한다면 어떻게 하겠는가. (Novorossiya 사례처럼 "흩어진 정보 연결" 예시를 들 수 있는가)
2. 온톨로지와 본인 KIFRS1115 프로젝트의 DB 스키마(ERD)는 무엇이 같고 무엇이 다른가.
3. 감사 업무에서 지식그래프로 풀 수 있는 문제를 하나 제시하고, 그 경우 인덱싱 비용을 감수할 가치가 있는지 어떻게 판단하겠는가.
