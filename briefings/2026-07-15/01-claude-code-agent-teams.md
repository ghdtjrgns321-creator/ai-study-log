# 오늘의 AI 개념: 클로드 코드 에이전트 팀즈(Agent Teams)

> 작성일: 2026-07-15 · 분류: claude-code

## 한 줄 정의

클로드 코드에서 여러 개의 독립된 클로드 세션이 하나의 팀을 이뤄, 서로 메시지를 직접 주고받으며 작업을 나눠 처리하는 기능이다.

## 쉬운 설명

기존 서브에이전트(sub-agent)는 메인 세션(리더)이 지시를 내리면 하위 에이전트가 작업 후 결과만 리더에게 보고하는 구조다. 하위 에이전트끼리는 서로 대화하지 못하고, 모든 소통은 리더를 거쳐야 한다.

에이전트 팀즈는 이 구조를 바꾼다. 팀 리더가 여러 명의 팀원(teammate)을 동시에 띄우면, 각 팀원은 자기만의 독립된 컨텍스트 창을 가지고 일하면서 다른 팀원에게 직접 메시지를 보낼 수 있다. 팀원들은 공유된 작업 목록(task list)에서 대기 중인 작업을 스스로 골라 가져간다.

회계법인의 감사팀에 비유하면 차이가 분명해진다. 서브에이전트 방식은 파트너가 시니어 한 명 한 명에게 개별적으로 지시하고 결과만 취합하는 구조에 가깝다. 에이전트 팀즈는 시니어 여러 명이 각자 담당 계정과목을 조사하다가 "이 거래 패턴이 이상하니 같이 확인해 달라"고 서로에게 직접 요청하고, 새 작업이 생기면 누구든 먼저 손을 든 사람이 맡는 구조에 가깝다.

두 방식의 근본적 차이는 통신 방향이다. 서브에이전트는 결과만 위로 올라가는 단방향 구조이고, 에이전트 팀즈는 팀원끼리 옆으로도 소통하는 양방향 구조다. 이 차이 때문에 에이전트 팀즈는 토큰 비용이 훨씬 크지만, 여러 관점이 서로를 검증해야 하는 작업에서는 품질 이득이 크다.

## 동작 원리

1. 리더 세션에서 사용자가 협업이 필요한 작업을 지시하면, 리더가 필요한 팀원 수와 역할을 판단해 팀원을 스폰한다.
2. 각 팀원은 별도의 클로드 코드 인스턴스로 실행되며, 프로젝트의 CLAUDE.md·MCP 서버·스킬은 동일하게 불러오지만 리더의 대화 이력은 물려받지 않는다.
3. 리더는 공유 작업 목록에 작업 항목을 만들고, 팀원은 이를 직접 배정받거나 스스로 다음 작업을 선점(claim)한다. 작업 간 의존관계가 있으면 선행 작업이 끝나야 후행 작업이 잠금 해제된다.
4. 팀원 간 메시지는 에이전트별 메일박스(JSON 파일)를 통해 자동 전달되며, 리더가 폴링할 필요가 없다.
5. 권한 승인은 항상 리더 세션에서 이뤄지며, 한 팀원이 거부당한 요청을 다른 팀원을 통해 우회할 수 없도록 설계돼 있다.

## 구체 예시·사례

풀 리퀘스트(PR) 리뷰가 대표 사례다. 리더에게 "보안·성능·테스트 커버리지를 각각 담당하는 팀원 세 명을 스폰해 PR 142를 리뷰하라"고 지시하면, 세 팀원이 동시에 같은 PR을 서로 다른 관점으로 검토하고 리더가 결과를 종합한다.

앤트로픽은 2026년 3월 이런 멀티에이전트 병렬 리뷰 방식을 자사 코드리뷰 기능(Claude Code Review)에 실전 적용했다고 밝혔다. 사내 테스트에서 상세한 리뷰 코멘트가 달린 PR 비율이 기존 16%에서 54%로 늘었고, 1000줄이 넘는 대형 PR의 84%에서 실제 문제가 발견됐다고 한다.

## 비슷한 것과 비교

| 구분 | 서브에이전트 | 에이전트 팀즈 | Git worktree 수동 병렬 |
|---|---|---|---|
| 통신 | 결과만 리더에게 보고 | 팀원끼리 직접 메시지 | 사람이 직접 조율 |
| 컨텍스트 | 리더에게 요약되어 귀속 | 완전히 독립 | 세션별로 완전히 독립 |
| 토큰 비용 | 낮음 | 높음(팀원 수만큼 배로) | 세션 수만큼 |
| 적합한 작업 | 결과만 필요한 단발성 조사 | 상호 검증·토론이 필요한 복잡한 작업 | 사람이 직접 감독하려는 작업 |

선택 기준은 한 줄로 정리된다. 팀원 간 대화와 상호 반박이 결과 품질을 좌우한다면 에이전트 팀즈를, 결과만 필요하면 서브에이전트를 쓴다.

## 왜 지금 중요한가

에이전트 팀즈는 2026년 2월 Opus 4.6 모델 출시와 함께 실험적 리서치 프리뷰로 공개됐다. 이후 v2.1.178에서 별도 설정 단계 없이 팀원을 바로 스폰할 수 있게 개선됐고, v2.1.199에서는 유휴 팀원 표시 방식이 정교해지는 등 지속적으로 업데이트되고 있다.

2026년 3월에는 이 병렬 협업 방식이 앤트로픽 자체 코드리뷰 기능에 실전 적용되며 구체적 성과 수치(리뷰 커버리지 16%→54%)로 검증됐다. 이는 실험 단계 기능이 사내 프로덕션에 적용되기까지 걸린 기간이 한 달 남짓이라는 뜻이어서, 멀티에이전트 협업 방식에 대한 실무 신뢰도가 빠르게 높아지고 있음을 보여준다.

## 회계법인 AI 직무 연결 포인트

감사 실무에서는 매출채권·재고자산·충당부채처럼 성격이 다른 계정과목을 동시에 검토해야 하는 경우가 많다. 에이전트 팀즈 구조를 적용하면 계정과목별로 팀원을 배정하고, 서로 다른 계정에서 발견한 이상 징후를 팀원끼리 교차 공유하게 만들어 개별 검토로는 놓치기 쉬운 계정 간 연관성(예: 매출 조기인식과 매출채권 증가의 동시 발생)을 포착할 수 있다.

문서에서 소개한 "경쟁 가설 조사" 패턴은 부정 조사 절차와 구조가 유사하다. 특정 거래의 이상 원인이 단순 오류인지, 고의적 부정인지, 시스템 결함인지 여러 팀원이 각자 가설을 세우고 서로 반박하며 좁혀가는 방식은, 감사인이 회의적 태도(professional skepticism)를 유지하며 여러 가능성을 배제해 나가는 절차를 소프트웨어로 흉내낸 것이라 설명할 수 있다.

다만 에이전트 팀즈는 토큰 비용이 서브에이전트보다 훨씬 크다. 면접에서는 이 기술을 무조건 도입해야 한다고 말하기보다, 상호 검증이 실제로 감사 품질을 높이는 고위험·고판단 영역(부정 위험 평가, 복잡한 추정치 검토)에 선별 적용해야 한다는 비용 대비 효익 관점을 함께 언급하는 것이 바람직하다.

## 핵심 용어·논쟁

- 팀 리더(Team Lead) — 팀원을 스폰하고 작업을 조율하는 메인 세션.
- 팀원(Teammate) — 독립된 컨텍스트에서 일하며 다른 팀원과 직접 통신하는 개별 클로드 세션.
- 작업 목록(Task List) — 팀원들이 상태(대기·진행·완료)를 공유하며 스스로 작업을 가져가는 목록.
- 메일박스(Mailbox) — 에이전트 간 메시지를 전달하는 JSON 파일 기반 통신 채널.

이 기능은 아직 실험적(experimental) 단계로, 세션 재개(`/resume`) 시 팀원이 복원되지 않는 등 한계가 명시돼 있다. 따라서 "병렬화하면 항상 빠르고 좋다"는 통념과 달리, 토큰 비용과 조율 오버헤드를 고려해 팀 규모와 작업 적합성을 판단해야 한다는 것이 실무자들 사이의 공통된 지적이다.

## 자료 깊이 읽기

### Orchestrate teams of Claude Code sessions — 영어, 공식문서, 중급
클로드 코드 공식 문서로, 에이전트 팀즈의 활성화 방법부터 아키텍처(팀 리더·팀원·작업 목록·메일박스)까지 가장 정확하게 설명한다. 서브에이전트와의 차이를 표로 비교하고, 언제 에이전트 팀즈를 쓰고 언제 쓰지 말아야 하는지 구체적 기준을 제시한다. PR 리뷰, 경쟁 가설 조사 등 실전 프롬프트 예시가 포함돼 있어 바로 따라 해볼 수 있다. 토큰 비용, 팀 규모 권장치(3~5명), 작업당 적정 개수(5~6개) 등 운영 팁도 담겨 있다. 실험적 기능의 알려진 한계(세션 재개 미지원, 중첩 팀 불가 등)를 투명하게 공개하고 있어 신뢰도가 높다.
