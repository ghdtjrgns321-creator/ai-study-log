# 오늘의 AI 개념: 에이전틱 RAG(Agentic RAG)

> 작성일: 2026-07-06 · 분류: ai-concept

## 한 줄 정의

AI가 자료를 한 번만 검색하고 끝내는 대신, 검색 결과가 부족하면 스스로 판단해서 다시 찾아보고 답을 다듬어 가는 방식이다.

## 쉬운 설명

검색증강생성(RAG, Retrieval-Augmented Generation)은 원래 질문 하나에 대해 관련 문서를 한 번 검색하고, 그 결과를 근거로 답을 생성하는 구조다. 에이전틱 RAG는 여기에 AI 에이전트의 자율 판단을 결합한 방식으로, 검색 결과가 질문과 맞지 않거나 부족하면 검색어를 바꾸거나 다른 자료원을 시도하고, 필요하면 검색과 답변 생성을 여러 차례 반복한다.

비유하자면 기존 RAG는 도서관 사서에게 책 한 권을 받아 오는 일에 가깝다. 에이전틱 RAG는 그 책이 질문과 맞지 않을 때 다른 서가를 뒤지거나 다른 사서에게 다시 물어보는 등, 스스로 판단해 조사를 이어가는 조사원에 가깝다.

기존 RAG와의 차이는 검색 여부와 검색 대상을 누가 결정하느냐에 있다. 기존 RAG는 미리 정해진 파이프라인을 따라 한 번 검색하고 끝나지만, 에이전틱 RAG는 검색이 더 필요한지, 계산기·사내 데이터베이스·외부 API 중 어떤 도구를 쓸지를 모델이 그때그때 판단한다.

## 동작 원리

에이전틱 RAG의 핵심은 단발 검색을 **"검색 → 평가 → 재검색 → 검증"의 반복 루프**로 바꾸는 것이다. 에이전트는 검색 결과를 그대로 쓰지 않고, 질문에 충분한지 스스로 평가한 뒤 부족하면 검색어를 바꿔 다시 찾고, 최종 답을 내기 전에 근거를 검증한다.

이 판단을 실제로 실행하는 장치가 **함수 호출(function calling)** 이다. Claude·GPT·Llama 같은 모델에 미리 정의한 도구(사내 DB 조회, 웹 검색, 계산기 등)를 붙여 두면, 모델이 "지금 어떤 도구를 쓸지"를 스스로 골라 호출한다.

구조는 규모에 따라 둘로 나뉜다.

- **단일 에이전트(라우터)** — 하나의 에이전트가 여러 지식원(DB·API 등) 중 어디를 조회할지 고르는 라우터 역할만 한다. 가장 단순한 형태다.
- **다중 에이전트** — 마스터 에이전트가 여러 특화 에이전트를 조율한다. 각자 내부 데이터·개인 계정·웹 검색 등 서로 다른 검색을 맡는다.

정리하면 바닐라 RAG가 "벡터 검색 한 번 → 생성"인 데 비해, 에이전틱 RAG는 다단계 검색·질의 전처리·결과 검증·외부 도구 사용을 포함하는 더 견고한 파이프라인이다.

## 구체 예시·사례

감사 증거 수집 상황을 따라가 보자. 회계사가 "이 거래가 이상한가?"를 질의하면, 에이전틱 RAG 시스템은 (1) 먼저 **총계정원장**을 검색해 거래 내역을 확인하고, (2) 결과가 판단에 불충분하면 **내부 정책 문서**를 추가 검색해 해당 거래 유형의 규정을 찾고, (3) 그래도 근거가 약하면 **과거 감사조서**로 검색을 넓혀 유사 선례를 붙인다. 각 단계마다 "지금 근거가 충분한가"를 에이전트가 평가해 다음 검색 여부를 정한다. 첫 검색 한 번으로 끝내는 바닐라 RAG였다면 총계정원장 결과만 보고 답을 냈겠지만, 에이전틱 RAG는 근거가 찰 때까지 검색 대상을 스스로 확장한다.

## 비슷한 것과 비교

| 항목           | 바닐라 RAG      | 에이전틱 RAG               |
| -------------- | --------------- | -------------------------- |
| 검색 횟수      | 1회(단발)       | 필요 시 반복               |
| 검색 대상 결정 | 고정 파이프라인 | 모델이 그때그때 판단       |
| 결과 검증      | 없음(바로 생성) | 평가·재검색·검증 루프      |
| 도구 사용      | 벡터 검색만     | DB·API·계산기 등 다중 도구 |
| 비용·지연      | 낮음·예측 가능  | 높음·가변                  |

선택 기준: 질의가 단순하고 자료원이 하나면 바닐라 RAG로 충분하다. 여러 시스템에 자료가 흩어져 있고 한 번 검색으로 답이 안 나오는 복합 질의면 에이전틱 RAG가 맞다.

## 왜 지금 중요한가

LangChain의 2026년 State of Agent Engineering 리포트에 따르면, 설문에 응답한 조직의 57.3%가 이미 AI 에이전트를 프로덕션 환경에서 운영 중이며, 이는 전년도 51%보다 늘어난 수치다.

시장조사기관 Gartner는 2026년 말까지 전체 기업용 애플리케이션의 40%가 특정 작업에 특화된 AI 에이전트를 탑재할 것으로 전망했다. 이는 2025년의 5% 미만에서 큰 폭으로 늘어난 수치다.

검색과 생성을 고정된 파이프라인으로 묶어 두는 대신 에이전트가 검색 전략을 스스로 조정하는 에이전틱 RAG는, 이러한 에이전트 확산의 핵심 구성 요소로 최근 논의되고 있다.

## 회계법인 AI 직무 연결 포인트

감사 증거 수집 과정에서 특정 거래의 이상 여부를 질의하면, 에이전틱 RAG 시스템은 총계정원장·내부 정책 문서·과거 감사조서를 순차적으로 검색하고, 첫 검색 결과가 불충분하면 검색 범위를 넓혀 재검색하도록 만들 수 있다.

표본 추출 기반 감사에서 전수 데이터를 상시 점검하는 지속적 감사(continuous auditing)로 흐름이 이동하는 가운데, 에이전틱 RAG는 이상 거래를 탐지한 뒤 관련 규정과 과거 사례를 자동으로 찾아 근거를 붙이는 역할을 맡을 수 있다.

ERP(Enterprise Resource Planning, 전사적 자원관리), 회계 시스템, 규정 문서가 여러 시스템에 분산된 회계법인 환경에서는 한 번의 검색으로 복합 질의를 처리하기 어렵다. 에이전틱 RAG는 어느 데이터 원천을 검색할지 스스로 판단해 검색 대상을 순차적으로 넓혀 가므로 이런 상황에 대응할 수 있다.

## 핵심 용어·논쟁

- **바닐라 RAG(Vanilla RAG)** — 한 번 검색하고 그 결과로 바로 답을 생성하는 기본형 RAG.
- **라우터(Router)** — 여러 지식원 중 어디를 조회할지 고르는 역할. 단일 에이전트 구조의 핵심.
- **함수 호출(Function calling)** — 모델이 미리 정의된 도구(DB·API 등)를 스스로 골라 호출하는 기능. 에이전트의 자율 검색을 실제로 실행하는 장치.
- **다중 에이전트(Multi-agent)** — 마스터 에이전트가 특화 에이전트들을 조율해 서로 다른 검색을 분담하는 구조.

논쟁 지점: 검색을 반복할수록 근거는 두터워지지만, 잘못된 중간 결과를 다음 검색의 전제로 삼으면 오히려 틀린 결론으로 미끄러질 수 있다. 또 반복 검색은 비용과 지연을 가변적으로 키운다. 언제 검색을 멈출지(종료 조건)와 중간 결과를 어떻게 검증할지가 실무 통제의 핵심이다.

## 자료 깊이 읽기

### What is Agentic RAG? (Weaviate) — 영어 / 텍스트 / 중급
에이전틱 RAG는 RAG 파이프라인에 AI 에이전트를 넣어 "계획·도구 사용·검증 루프"를 더함으로써 신뢰성을 높인 방식이다. 단발 조회 대신 검색→평가→재검색→검증을 반복해 최종 답을 낸다. 가장 단순한 단일 에이전트 형태에서는 에이전트가 라우터로서 여러 외부 지식원(DB·API 등) 중 어디를 조회할지 고른다. 다중 에이전트 형태에서는 마스터 에이전트가 특화 에이전트들을 조율해 내부 데이터·개인 계정·웹 검색 등을 나눠 맡는다. GPT-4·Claude·Llama 같은 모델에 미리 정의한 도구를 붙여 함수 호출로 외부 서비스·DB와 상호작용하게 한다. 바닐라 RAG와 달리 다단계 검색·질의 전처리·결과 검증·외부 도구 사용을 지원해, 복합 질의를 더 견고하게 처리하고 특화된 라우팅·검증 계층으로 정확도를 높인다.
