# 오늘의 AI 개념: MCP 도구 호출 자동 백그라운드 전환(Automatic MCP Tool Backgrounding)

> 작성일: 2026-07-22 · 분류: claude-code

## 한 줄 정의

클로드 코드가 2분 넘게 걸리는 MCP(모델 컨텍스트 프로토콜, Model Context Protocol) 도구 호출을 세션을 멈추지 않고 자동으로 백그라운드로 넘기는 기능이다.

## 쉬운 설명

기존에는 클로드 코드가 MCP 도구를 호출하면 응답이 올 때까지 대화 전체가 멈췄다. 예를 들어 주피터 노트북 셀에서 모델 학습을 10분간 돌리면, 그 10분 동안 사용자는 다른 지시를 내릴 수도, 다른 결과를 확인할 수도 없었다.

이는 전화로 은행 창구 직원에게 서류 처리를 요청했을 때, 처리가 끝날 때까지 수화기를 들고 아무것도 못 한 채 기다리는 상황과 같다. 이번 업데이트는 "처리되면 콜백 주세요"라고 말하고 전화를 끊은 뒤 다른 업무를 보다가, 알림이 오면 다시 확인하는 방식으로 바꾼 것이다.

기존에도 서브에이전트를 백그라운드로 돌리는 기능은 있었지만(2026-07-06 항목 참고), 그것은 "하위 작업 전체"를 병렬로 분리하는 개념이다. 이번 기능은 그와 달리 "메인 세션 안에서 실행 중인 개별 MCP 도구 호출 한 건"이 예상보다 오래 걸릴 때, 그 호출 하나만 비동기로 전환한다는 점에서 대상 단위가 다르다.

## 동작 원리

1. 클로드가 메인 대화 세션에서 MCP 도구를 호출한다.
2. 응답 시간이 `CLAUDE_CODE_MCP_AUTO_BACKGROUND_MS`(기본값 2분)를 넘기면, 해당 호출이 백그라운드 태스크로 자동 전환된다.
3. 클로드는 태스크 ID를 즉시 받아 세션을 막지 않고 다른 작업을 계속한다.
4. 진행 중인 태스크는 `/tasks`에서 확인·중단할 수 있다.
5. 호출이 끝나면 결과가 "태스크 알림"으로 세션에 전달된다.

이 기능은 클로드 코드 v2.1.212 이상에서 동작하며, 임계값을 조정하거나 `0`으로 설정해 기능 자체를 끌 수 있다. 다만 백그라운드 태스크는 세션이 끝나면 함께 사라지므로, 세션을 넘어 결과를 보존해야 하는 작업에는 별도 저장이 필요하다.

## 구체 예시·사례

감사팀이 MCP로 연결된 사내 데이터 웨어하우스에서 매출채권 이상거래 탐지 모델을 재학습시키는 상황을 가정한다. 학습 셀 실행에 10분이 걸린다면, 기존 방식에서는 그 10분 동안 감사인이 클로드 코드로 다른 감사조서를 검토하거나 질의할 수 없었다.

이번 업데이트가 적용되면 2분 시점에 호출이 자동으로 백그라운드 태스크로 전환되고, 감사인은 태스크 ID를 받은 즉시 다른 작업으로 넘어갈 수 있다. 학습이 끝나면 알림이 도착하고, `/tasks`에서 결과를 확인하면 된다.

## 비슷한 것과 비교

| 구분 | 백그라운드 서브에이전트(2026-07-01, Week 27) | MCP 자동 백그라운드(이번 업데이트, Week 29) |
|------|------|------|
| 전환 대상 | 서브에이전트 프로세스 전체 | 개별 MCP 도구 호출 1건 |
| 전환 시점 | 기본적으로 항상 백그라운드 실행 | 응답이 2분을 넘을 때만 자동 전환 |
| 확인 위치 | `claude agents`(에이전트 뷰) | `/tasks` |
| 목적 | 여러 하위 작업을 나눠 병렬 처리 | 느린 MCP 호출 한 건 때문에 세션이 멈추는 것을 방지 |

여러 작업을 나눠 병렬로 처리하려면 서브에이전트를, 특정 MCP 도구 호출이 가끔 오래 걸려 세션이 멎는 상황을 막으려면 이번 자동 백그라운드 전환을 쓴다고 정리할 수 있다.

## 왜 지금 중요한가

이 기능은 클로드 코드 v2.1.207~v2.1.212(Week 29, 2026년 7월 13~17일)에 공식 반영됐다. 같은 주간에 세션 전역 캐핑(WebSearch 호출·서브에이전트 스폰 기본 200회 제한)도 함께 도입돼, 에이전트가 장시간·고빈도 작업에서 폭주하지 않도록 안전장치를 강화하는 흐름과 맞닿아 있다.

이 문제의식은 애초에 깃허브 이슈 #18617(SEP-1686 MCP Task Mode 지원 요청)에서 "MCP 도구 호출이 동기적이라 장시간 작업 시 전체 대화가 막힌다"는 지적으로 제기됐던 사안이다.

## 회계법인 AI 직무 연결 포인트

감사·세무 업무에서 MCP로 연결한 ERP·데이터 웨어하우스 조회, 대용량 XBRL 파싱, 야간 배치성 이상거래 탐지 같은 작업은 실행 시간이 들쭉날쭉하다. 이런 호출이 자동으로 백그라운드로 넘어가면, 감사인은 대기 없이 다른 절차를 병행할 수 있어 감사 인력의 유휴 시간이 줄어든다.

지속적 감사(Continuous Auditing, 2026-07-11 항목) 파이프라인처럼 상시로 데이터를 조회·분석하는 구조에서는 개별 조회 하나가 오래 걸려도 전체 워크플로우가 막히지 않는 것이 중요한데, 이번 기능이 그 지점을 직접 보완한다.

다만 백그라운드 태스크가 세션 종료와 함께 사라진다는 점은, 감사 증거로 남겨야 하는 장시간 작업 결과를 별도 로그나 감사조서 시스템에 영속적으로 저장하는 절차를 함께 설계해야 한다는 실무적 유의점으로 이어진다.

## 핵심 용어·논쟁

- **MCP(Model Context Protocol)** — LLM이 외부 도구·데이터소스에 표준화된 방식으로 연결하는 프로토콜.
- **블로킹(Blocking) 호출** — 응답이 돌아올 때까지 전체 프로세스가 멈추는 호출 방식.
- **태스크 알림(Task Notification)** — 백그라운드 작업이 끝났을 때 세션에 전달되는 알림.
- **세션 전역 캐핑** — 같은 주간 업데이트에 포함된 기능으로, WebSearch 호출·서브에이전트 스폰을 세션당 기본 200회로 제한해 폭주를 막는 장치.

이 기능 자체는 진행 중인 찬반 논쟁이 있는 주제라기보다, 동기 호출의 구조적 한계를 보완하는 실무 편의 기능에 가깝다.
