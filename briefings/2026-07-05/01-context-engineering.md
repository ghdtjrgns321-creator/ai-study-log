# 오늘의 AI 개념: 컨텍스트 엔지니어링(Context Engineering)

> 작성일: 2026-07-05 (2026-07-24 심화 개편) · 분류: ai-concept

## 한 줄 정의

AI에게 질문 문구만 잘 다듬어 주는 대신, 좋은 답을 내는 데 필요한 배경 정보 전체를 체계적으로 준비해서 넣어 주는 작업이다.

## 쉬운 설명

컨텍스트 엔지니어링은 거대언어모델(LLM, Large Language Model)이 특정 작업을 수행할 때 참고할 지식, 이전 대화 기록, 회사 규정, 외부 문서, 도구 사용 결과 등 필요한 정보 전체를 설계하고 관리하는 작업을 말한다. 단발성 질문 하나를 잘 쓰는 프롬프트 엔지니어링과 달리, 여러 단계와 여러 세션에 걸쳐 어떤 정보를 언제 모델에 넣어 줄지 시스템 차원에서 설계한다.

비유하자면 프롬프트 엔지니어링이 신입사원에게 오늘 할 업무 지시를 명확하게 전달하는 일이라면, 컨텍스트 엔지니어링은 그 신입사원이 일을 제대로 하도록 사규집, 이전 업무 이력, 관련 부서 연락처, 참고 서식을 미리 책상 위에 챙겨 놓아 주는 일에 가깝다. 질문을 잘 만드는 기술이 아니라, 질문에 답하는 데 필요한 재료를 빠짐없이 갖춰 주는 기술이다.

프롬프트 엔지니어링과 완전히 대체 관계는 아니다. 프롬프트 엔지니어링은 단일 지시문의 표현을 다듬는 좁은 범위의 작업이고, 컨텍스트 엔지니어링은 RAG(검색해 온 자료를 근거로 답변을 생성하는 기법), 장단기 기억, 도구 연결까지 포함하는 더 넓은 범위의 시스템 설계 작업이다.

## 동작 원리

출발점은 "컨텍스트는 유한 자원"이라는 전제다. 트랜스포머 구조상 모든 토큰이 서로를 참조하므로(토큰 n개면 쌍 관계 n²개) 컨텍스트가 길어질수록 주의(attention) 예산이 분산되고, 넣어 준 정보를 되찾는 정확도가 떨어지는 컨텍스트 부패(context rot)가 생긴다. 그래서 설계 원칙은 "원하는 결과를 낼 확률을 최대화하는, 가장 작은 고신호(high-signal) 토큰 집합 찾기"가 된다.

```
후보 정보(지시문·문서·대화 이력·도구 출력·메모리)
   │ ①선별: 적정 고도의 시스템 프롬프트 · 중복 없는 도구 구성 ·
   │        식별자(경로·ID)만 두고 필요할 때 로드(저스트인타임 검색)
   ▼
컨텍스트 창(유한 예산) ─▶ LLM 추론 ─▶ 출력·도구 호출
   ▲ ②창이 차오르면: 압축(요약 후 재시작) · 외부 노트 기록 ·
   │                  서브에이전트 위임(요약만 회수)
   └────────────────────────────────────────┘
```

①이 "무엇을 넣을까", ②가 창보다 오래 걸리는 장기 작업에서 "어떻게 버틸까"의 기술이다. 두 축을 합쳐 매 추론 시점마다 창 안에 고신호 정보만 유지되도록 순환시킨다.

## 구체 예시·사례

코딩 에이전트(Claude Code, Cursor 등)가 대규모 리팩터링을 수행하는 상황을 따라가 보자.

1. **선별해서 넣기** — 프로젝트 규칙 파일(예: CLAUDE.md)이 세션마다 자동 주입돼 영속 메모리 역할을 한다. 코드베이스 전체 대신 파일 경로 목록만 유지하고, 실제 내용은 수정할 때 그때그때 읽는다(저스트인타임 검색).
2. **버티기** — 수십 개 파일을 고치다 창이 한계에 다가가면, 지금까지의 결정 사항·남은 작업을 요약해 새 컨텍스트로 넘긴다(압축). 동시에 진행 상황을 노트 파일에 적어 두어, 리셋돼도 이어받는다(노트 작성).
3. **위임** — "테스트 전부 실행하고 실패 원인만 보고해" 같은 하위 작업은 서브에이전트에 맡긴다. 수천 줄의 로그는 서브에이전트의 컨텍스트에서 소화되고, 본체에는 실패 3건의 요약만 돌아온다.

같은 모델·같은 프롬프트라도 이 설계 유무에 따라 장시간 작업의 완주율이 달라진다는 것이 컨텍스트 엔지니어링의 요지다.

## 왜 지금 중요한가

에이전트형 AI가 여러 단계를 스스로 계획하고 실행하는 구조로 발전하면서, 단일 프롬프트만으로는 일관된 결과를 내기 어려워졌다. Anthropic은 공식 엔지니어링 블로그에서 질문 자체가 "좋은 프롬프트 쓰기"에서 "어떤 컨텍스트 구성이 원하는 행동을 낳는가"로 이동했다고 정리하고, 컨텍스트를 유한 자원으로 다루는 설계 전략을 제시했다.

세무·회계 분야에서도 같은 논의가 나온다. Thomson Reuters는 신뢰할 수 있고 감사 가능한 결과를 얻으려면 질문을 잘 던지는 것을 넘어, 권위 있는 근거 출처·시행일·고객 이력 같은 맥락 정보를 AI에 체계적으로 제공해야 한다고 설명한다.

- [Effective context engineering for AI agents (Anthropic)](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)
- [What context engineering means for tax and accounting (Thomson Reuters)](https://tax.thomsonreuters.com/blog/context-engineering-in-tax-and-accounting-the-next-level-of-ai-workflows/)

## 회계법인 AI 직무 연결 포인트

감사 업무에서는 회사별 조직 구조, 계정과목 정의, 표준운영절차(SOP)를 컨텍스트로 미리 구성해 두면, AI가 여러 회사·여러 기간의 감사 업무에서 일관된 판단 기준을 적용하도록 만들 수 있다. Thomson Reuters가 꼽는 필수 컨텍스트 요소 — 권위 출처, 규정 시행일, 고객·펌 이력, 추론 투명성, 한계 인지 — 는 그대로 감사조서의 품질 요건과 겹친다.

RAG와 결합하면 회계기준서, 과거 감사조서, 재무제표 주석을 컨텍스트로 관리해 AI가 답변마다 근거 문서를 함께 제시하도록 만들 수 있다. 파트너 리뷰를 통과하는 산출물과 컴플라이언스 리스크가 되는 산출물을 가르는 것이 결국 컨텍스트 설계라는 것이 Thomson Reuters의 논지다.

모델 컨텍스트 프로토콜(MCP, Model Context Protocol) 같은 표준을 활용하면 ERP나 회계 시스템의 실시간 데이터를 컨텍스트로 연결할 수 있어, 분개 내역·잔액 조회 같은 최신 데이터를 반영한 분석이 가능해진다.

## 핵심 용어·논쟁

- **컨텍스트 창(Context Window)** — 모델이 한 번의 추론에서 볼 수 있는 토큰 예산. 유한 자원.
- **컨텍스트 부패(Context Rot)** — 컨텍스트가 길수록 넣어 둔 정보의 회상 정확도가 떨어지는 현상.
- **압축(Compaction)** — 창 한계에 다가가면 대화를 요약해 새 컨텍스트로 넘기는 기법.
- **저스트인타임 검색(Just-in-Time Retrieval)** — 식별자만 유지하다가 실행 중 필요한 것만 로드하는 전략.
- **서브에이전트(Sub-agent)** — 하위 작업을 별도 컨텍스트에서 수행하고 요약만 반환하는 보조 에이전트.

쟁점은 "어디까지 미리 넣고, 어디서부터 실행 중에 찾게 할 것인가"다. 사전 로드는 빠르지만 창을 낭비하고, 저스트인타임은 창을 아끼지만 탐색 시간이 든다. 정답은 작업 성격에 따른 혼합 전략이며, 이 배분 감각이 실무 역량으로 꼽힌다.

## 자료 깊이 읽기

### Effective context engineering for AI agents (Anthropic 엔지니어링 블로그) — 영어 / 텍스트 / 중급
이 주제의 1차 자료. 트랜스포머의 n² 쌍 관계 때문에 컨텍스트가 길수록 주의 예산이 고갈되는 컨텍스트 부패를 근거로, 컨텍스트를 유한 자원으로 다루라고 주장한다. 시스템 프롬프트는 경직된 하드코딩과 모호한 지시 사이의 적정 고도를 찾고, 도구는 기능 중복 없이 최소로 구성하며, 데이터는 식별자만 두고 저스트인타임으로 로드하라고 권한다. 장기 작업용으로 압축·노트 작성·서브에이전트 세 기법을 제시하며, 총괄 원칙은 "결과를 최대화하는 가장 작은 고신호 토큰 집합"이다.

### What context engineering means for tax and accounting (Thomson Reuters) — 영어 / 텍스트 / 중급
컨텍스트 엔지니어링을 "방대한 정보에서 의사결정으로 이어지는 가장 작은 고신호 입력 집합을 큐레이션하는 일"로 정의하고, 프롬프트 엔지니어링과 달리 여러 상호작용·세션·사용자에 걸친 정보 환경 관리라고 구분한다. 세무·회계 AI의 필수 컨텍스트로 권위 출처, 규정 시행일, 고객·펌 이력, 추론 투명성, 한계 인지 다섯 가지를 제시하고, 감사 추적과 인용 검증이 가능한 산출물만이 파트너 리뷰를 통과한다고 강조한다.

### Context Engineering: AI 시대의 새로운 핵심 역량 (SK 데보션) — 한국어 / 텍스트 / 중급
구성 요소를 시스템 지시문·대화 이력과 메모리·지식 통합(검색 문서, 도구 출력) 세 축으로 정리하고, 기법으로 요청별 동적 조립, 관련성 기반 우선순위, 토큰 예산 관리, 계층적 요약을 든다. Cursor와 Claude Code가 코드베이스 인덱싱과 CLAUDE.md 영속 메모리로 이를 구현한 실례를 보여주며, "프롬프트는 무엇을 말할지, 컨텍스트는 모델이 무엇을 볼 수 있을지를 설계한다"로 차이를 요약한다.

## 자가 점검 질문

1. 프롬프트 엔지니어링과 컨텍스트 엔지니어링의 차이를 한 문장으로 설명한다면 무엇인가?
2. 감사 업무에 컨텍스트 엔지니어링을 적용할 때, 어떤 사내 문서나 데이터를 컨텍스트로 우선 구성해야 하는가?
3. 컨텍스트를 과도하게 많이 제공했을 때 발생할 수 있는 문제(컨텍스트 부패)는 무엇이며, 압축·노트·서브에이전트 중 어떤 기법으로 관리할 수 있는가?
