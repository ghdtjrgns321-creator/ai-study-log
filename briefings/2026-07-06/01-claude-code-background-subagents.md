# 오늘의 AI 개념: 백그라운드 서브에이전트와 알림 훅(Background Subagents & Notification Hooks)

> 작성일: 2026-07-06 · 분류: claude-code

## 한 줄 정의

Claude Code가 하위 작업을 맡은 서브에이전트를 메인 대화를 막지 않고 백그라운드에서 실행하고, 작업이 끝나거나 입력이 필요해지면 알림 훅으로 사용자에게 알려주는 기능이다.

## 쉬운 설명

기존에는 서브에이전트(Task 도구로 호출하는 하위 AI)를 부르면 메인 세션이 그 작업이 끝날 때까지 기다려야 했다. 2026년 7월 1일 배포된 버전(v2.1.198)부터는 서브에이전트가 기본적으로 백그라운드에서 실행되어, 메인 세션은 그동안 다른 작업을 계속하다가 서브에이전트가 끝나면 알림을 받는 방식으로 바뀌었다.

비유하자면, 예전에는 부장이 신입사원 한 명에게 조사를 시켜놓고 그 사람이 끝낼 때까지 팔짱 끼고 기다려야 했다면, 이제는 여러 명에게 동시에 일을 맡기고 자기 할 일을 하다가 보고가 들어오면 그때 확인하는 식이다. `/tasks` 명령으로 지금 백그라운드에서 뭐가 돌고 있는지 확인할 수 있고, `Ctrl+B`로 실행 중인 작업을 즉시 백그라운드로 보낼 수도 있다.

기존 서브에이전트 개념과의 차이는 실행 모델에 있다. Task 도구로 서브에이전트를 부르는 것 자체는 이전에도 가능했지만 기본은 동기 실행(끝날 때까지 대기)이었다. 이번 변화는 비동기·백그라운드를 기본값으로 바꾸고, 훅이라는 관측 지점을 표준화한 것이 핵심이다.

## 동작 원리

서브에이전트의 핵심은 **독립된 컨텍스트 창**이다. 메인 세션이 Task 도구로 서브에이전트를 부르면, 그 서브에이전트는 메인 대화 기록을 물려받지 않고 빈 컨텍스트에서 시작한다. 자기 작업만 수행한 뒤 **관련 결과만** 메인으로 돌려주고, 중간의 방대한 탐색 과정은 메인 컨텍스트를 오염시키지 않는다. 편향 없는 코드 리뷰나 대량 검색에 서브에이전트가 적합한 이유가 이 격리 때문이다.

실행 흐름은 다음과 같다.

1. 메인 세션이 Task 도구로 서브에이전트를 호출한다(또는 `.claude/agents/`의 커스텀 에이전트를 지정).
2. 서브에이전트가 **백그라운드에서** 돈다. 메인은 멈추지 않고 다른 일을 계속한다.
3. 서브에이전트가 **입력이 필요**해지면 `agent_needs_input` 이벤트로, **작업을 마치면** `agent_completed` 이벤트로 `Notification` 훅이 발동한다.
4. 훅은 알림 자체를 막거나 내용을 바꿀 수는 없고, 데스크톱 알림·Slack 전송 같은 **부수효과만** 붙인다.
5. 사용자는 `/tasks`로 현재 돌아가는 작업을 확인하고, `Ctrl+B`로 실행 중 작업을 즉시 백그라운드로 보낸다.

여러 서브에이전트를 서로 독립적인 작업에 동시에 띄우면, 순차 실행보다 빠르게 끝난다. 반대로 앞 단계 결과에 강하게 의존하는 작업이나 같은 파일을 동시에 고치는 작업에는 병렬 서브에이전트가 오히려 충돌을 부른다.

## 비슷한 것과 비교

| 구분      | 동기 서브에이전트(이전 기본) | 백그라운드 서브에이전트(현재 기본) |
| --------- | ---------------------------- | ---------------------------------- |
| 메인 세션 | 작업 끝날 때까지 대기        | 멈추지 않고 계속 진행              |
| 병렬성    | 사실상 순차                  | 여러 개 동시 실행                  |
| 완료 인지 | 반환값으로 즉시              | `agent_completed` 훅 알림          |
| 적합한 일 | 짧고 즉시 필요한 하위 작업   | 오래 걸리는 독립 조사·검색         |

선택 기준: **결과를 지금 당장 이어 써야 하면** 동기처럼 기다리고, **독립적이고 오래 걸리는 일**이면 백그라운드로 던지고 알림을 받는다.

## 왜 지금 중요한가

Claude Code 공식 changelog에 따르면 v2.1.198(2026년 7월 1일)에서 "Subagents now run in the background by default, so Claude keeps working while they run and is notified when they finish"라는 변경과 함께, 백그라운드 세션이 입력을 기다리거나 끝났을 때 `Notification` 훅이 `agent_needs_input`/`agent_completed` 이벤트로 발동되는 기능이 추가됐다. 바로 다음 날인 v2.1.199(7월 2일)에서는 슬래시 스킬을 이어붙여 최대 5개까지 한 번에 불러오는 스킬 스태킹 기능도 함께 들어왔다.

## 회계법인 AI 직무 연결 포인트

여러 계정과목별 조서 검토나 잔액 확인처럼 서로 독립적인 하위 작업을, 각각 서로 다른 서브에이전트에 병렬로 맡기고 하나가 끝날 때마다 알림을 받아 검토하는 워크플로우를 구성할 수 있다. 실제로 이 브리핑 자동화 자체가 그런 백그라운드 서브에이전트 방식으로 동작한다.

알림 훅을 사내 메신저(Slack·Teams)에 연동하면, 야간에 돌려둔 배치성 감사 스크립트(예: 전수 거래 이상탐지)가 끝났을 때 담당 회계사에게 바로 알려줄 수 있어 상시감사(continuous auditing) 체계와 자연스럽게 맞물린다.

다만 백그라운드 서브에이전트는 사전 승인된 권한만 자동으로 실행되고 나머지는 자동으로 거부되므로, 회계 데이터나 고객 정보에 접근하는 자동화를 설계할 때는 어떤 권한을 사전 승인할지, 그리고 그 승인 범위가 적절한지를 신중히 검토해야 한다.

## 핵심 용어·논쟁

- **서브에이전트(Subagent)** — Task 도구로 호출하는 하위 Claude 인스턴스. 독립된 컨텍스트 창을 가진다.
- **컨텍스트 격리(Context isolation)** — 서브에이전트가 메인 대화 기록 없이 빈 상태에서 시작해, 결과만 돌려주는 구조.
- **Notification 훅** — 특정 이벤트에 반응해 부수효과를 실행하는 관측 지점. 알림을 막거나 바꾸지는 못한다.
- **`agent_needs_input` / `agent_completed`** — 각각 서브에이전트가 입력을 기다릴 때·작업을 마쳤을 때 발동하는 이벤트 이름.

논쟁 지점: 백그라운드 자동 실행은 편리하지만, "사전 승인된 권한만 자동 실행"이라는 안전장치의 승인 범위를 어디까지 넓힐지가 실무 리스크다. 편의를 위해 권한을 넓게 열면 자동화가 민감 데이터에 손댈 여지가 커진다.

## 자료 깊이 읽기

### How and when to use subagents in Claude Code — 영어 / 텍스트 / 입문
서브에이전트는 독립된 컨텍스트 창을 가진 별도 Claude 인스턴스로, 작업을 처리한 뒤 관련 결과만 메인 대화로 돌려준다. 조사 중심 작업, 병렬 독립 작업, 신선한 시각이 필요한 검증, 단계가 뚜렷한 파이프라인에 적합하다. `Ctrl+B`로 백그라운드 전환, `/tasks`로 실행 현황 확인이 가능하다. 각 서브에이전트가 대화 기록에 얽매이지 않고 백지에서 시작하므로 편향 없는 코드 리뷰·객관적 분석에 유리하다. 호출 방식은 대화형 프롬프트, `.claude/agents/`의 커스텀 정의, CLAUDE.md 위임 정책, 스킬 등 여러 가지다. 반대로 순차 의존 작업, 같은 파일 동시 편집, 짧은 작업, 에이전트 간 협응이 필요한 경우엔 쓰지 않는 게 좋다.

### Claude Code changelog (v2.1.198 / v2.1.199) — 영어 / 텍스트 / 중급
v2.1.198(7월 1일)은 "서브에이전트가 이제 기본적으로 백그라운드에서 실행되어, Claude가 그동안 계속 작업하다가 끝나면 알림을 받는다"고 명시한다. 같은 버전에서 `claude agents`에 백그라운드 알림이 추가되어, 입력이 필요하거나 끝난 세션이 `Notification` 훅을 `agent_needs_input`/`agent_completed`로 발동한다. 이튿날 v2.1.199(7월 2일)에서는 `/skill-a /skill-b do XYZ`처럼 슬래시 스킬을 이어붙이면 맨 앞 스킬만이 아니라 앞쪽 스킬을 최대 5개까지 모두 로드하는 스킬 스태킹이 들어왔다.

## 자가 점검 질문

1. 서브에이전트가 "백그라운드 기본 실행"으로 바뀌면서 기존 동기 실행 방식과 무엇이 달라지는가?
2. 감사 업무에서 여러 계정과목 조서를 병렬 서브에이전트로 나눠 처리한다면, 어떤 순서·의존관계를 고려해야 하는가?
3. 알림 훅에 사내 데이터를 흘려보낼 때 어떤 보안·정보보호 리스크를 점검해야 하는가?
