# 영상 정리: 의료·제약 AI Engineer가 말하는 실무 RAG와 Agentic AI 전부 공개

> 원본: [의료·제약 AI Engineer가 말하는 실무 RAG와 Agentic AI 전부 공개](https://www.youtube.com/watch?v=ylJy6c-tt20) — 한국어, 유튜브, 중급, 약 55분(강연+실습)
> 연결된 브리핑: [2026-07-06.md](../../2026-07-06.md)
> 이 문서는 한국어 자막(약 2,000줄 분량)을 실제로 내려받아 확인하고 정리했다. 발표자는 서울대병원을 거쳐 현재 제약회사에서 LLM 서빙 인프라를 리드하는 5년차 AI 엔지니어이며, 청중은 주로 취업준비생이다. 강연 전반부(개념·아키텍처)와 후반부(실습 데모)로 구성되며, 이 문서는 개념 파트를 중심으로 정리한다.

## 강연이 던지는 핵심 메시지

발표자는 시작부터 이렇게 규정한다 — 취준생 입장에서 중요한 것은 "프롬프트를 잘 짜는 것"이 아니라, **내가 만든 결과물이 외부에서 호출 가능하고, 안정적으로 돌아가며, 문제가 생기면 추적이 가능한 형태인가**이다. 강연 전체가 이 한 문장을 향해 구성된다.

세 가지 핵심 주장:

1. 추론 API(안정적 인터페이스)가 모든 것의 시작이다.
2. 롱 컨텍스트·RAG·에이전트 같은 트렌드가 바뀌어도 운영 기본기는 똑같이 중요하다.
3. 계약(API 스펙)·로깅·도커 같은 실무 산출물을 어떻게 설명할지가 포트폴리오의 차별화 포인트다.

## 전체 파이프라인 조감도

```
클라이언트 ──HTTP 요청──▶ [FastAPI: 라우팅 + 입력 검증] ──▶ [LLM 추론 로직] ──▶ JSON 응답
```

발표자는 "이 단순한 그림이 면접에서 설명할 프로젝트의 뼈대"라고 강조한다. 단순히 "모델을 어떻게 썼다"가 아니라 "OpenAI 호환(compatible) 엔드포인트로 추론 서버를 제공했다 + Docker로 패키징했다 + Postman/Swagger로 API를 테스트했다 + 레이턴시를 측정했다"라고 말할 수 있어야 실무형으로 보인다는 것.

## 롱 컨텍스트 vs RAG — 기술 선택이 아니라 제품 의사결정

| 항목        | 롱 컨텍스트(Long Context)                                 | RAG                                                                                        |
| ----------- | --------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 구현 난이도 | 단순(문서를 통째로 넣음)                                  | 검색 로직 필요                                                                             |
| 강점        | 요약·전체 맥락 이해에 유리                                | 비용·속도 효율적, 최신 데이터 반영에 유리(재학습 불필요)                                   |
| 약점        | 입력 토큰 증가 → 비용·TTFT(Time To First Token) 지연 증가 | 검색 실패 시 답변 품질 급락, "로스트 인 더 미들"(긴 컨텍스트 중간 정보를 놓치는 현상) 위험 |

실무 결론: "좋은 컨텍스트를 비용·지연 제약 안에서 어떻게 넣느냐"가 핵심이며, 이를 위해 컨텍스트를 주입하는 파이프라인 자체를 설계해야 한다(단순히 "검색을 붙인다"는 발상으로는 부족).

## 컨텍스트 파이프라인 3단계

```
[Step 1: 검색] 벡터 DB·키워드 검색으로 후보 문서 확보
        │
        ▼
[Step 2: 프로세싱] 청크로 검색했더라도 반환은 문서 전체 단위로 확장,
                    또는 상위 20개 검색 후 5개로 리랭킹 — 노이즈 감소
        │
        ▼
[Step 3: 어셈블리] 템플릿으로 프롬프트를 만들고 컨텍스트를 주입
        │
        ▼
[Step 4: 추론(Inference)] ← 강연이 오늘 집중하는 지점
```

발표자가 **추론(Step 4)부터 고정**하는 이유: 추론 인터페이스가 안정적이면 앞단 1~3번 로직이 어떻게 바뀌어도(검색 단계가 빠지거나, 리랭킹을 두 번 하거나) 외부 사용자는 영향을 받지 않는다. 이것이 "한 번의 데모로 끝나지 않는 확장 가능한 구조"를 보여주는 포인트라고 강조한다.

또한 에이전트 시대에는 컨텍스트 입력이 문서뿐 아니라 **도구(API 스펙, 함수 설명)**와 **메모리(이전 대화, 사용자 선호)**까지 섞인다. Claude Code의 스킬(MD 파일로 API 스펙·함수 설명을 정리해두는 것)이 예시로 언급된다. 이렇게 섞이는 컨텍스트가 많아질수록 "왜 이런 답이 나왔는지" 추적하는 디버깅이 어려워지므로, **요청 ID(request ID)와 입력 로그**가 실무 필수 항목이 된다.

## LLM 엔지니어의 역할 — 3영역의 교차점

```
[머신러닝: 프롬프트·파인튜닝·평가]
              │
[백엔드 시스템: API·DB·도커·CI/CD]──┼──[프로덕트: 사용자 가치·UX·유즈케이스]
              │
       LLM 엔지니어 = 요구사항을 입출력 스펙으로 "번역"하고,
                      비용·속도 제약 안에서 구현하며,
                      정량 지표로 평가하는 사람(Translator & Builder)
```

## LLMOps — 랩 데모를 프로덕션으로 만드는 4단계 순환

```
[Build] 모델 선택·파인튜닝·프롬프트 엔지니어링
    │
    ▼
[Deploy] 컨테이너화 + API 엔드포인트 노출
    │
    ▼
[Monitor] 로그·메트릭 수집 → 레이턴시·비용 추적
    │
    ▼
[Evaluate & Improve] A/B 테스트, 사용자 피드백/로그 반영, 재학습·프롬프트 개선
    │
    └──────────────▶ 다시 Build로 순환
```

**LLMOps 최소 구성요소 5가지(실무 체크리스트)**:

1. **버전 컨트롤**: 모델·프롬프트를 버전으로 관리.
2. **평가(Evaluation)**: 자동 평가 파이프라인 구축, 골든 스탠더드 데이터셋으로 벤치마크(MMLU, SWE-bench 등 목적에 맞는 벤치마크 활용).
3. **배포(Deployment)**: Docker로 패키징, CI/CD로 배포 자동화. 호스트 시스템 의존성에서 자유로운 것이 Docker를 쓰는 이유.
4. **모니터링**: 로그·레이턴시를 대시보드로 추적하고 알림 수신. Langfuse(오픈소스, 셀프호스팅 가능)·LangSmith가 실무에서 널리 쓰이는 프레임워크로 언급된다.
5. **피드백 루프**: 사용자 피드백을 근거로 배드 케이스(bad case)를 분석 — 예: RAG가 정확한 문서를 반환하지 못해 맥락이 없어 답변이 부실한 경우, 또는 에이전트가 적절한 툴을 호출하지 못하는 경우.

포트폴리오 팁: 거창한 MLOps 플랫폼이 아니어도 된다. 이미지에 `v0.0.1` 같은 버전 태그를 붙여 Git에 주기적으로 올리거나, 매 요청마다 request ID 로그를 남겨 추적 가능성을 열어두는 정도로도 "운영을 안다"는 것을 산출물로 증명할 수 있다.

## 실무에서 반드시 부딪히는 5대 이슈와 대응

| 이슈              | 증상                                                               | 대응                                                                                   |
| ----------------- | ------------------------------------------------------------------ | -------------------------------------------------------------------------------------- |
| 레이턴시          | 응답 5초 지연 시 사용자 이탈 급증(웹은 3초 로딩 지연에 90%+ 이탈)  | 타임아웃 설정, 모델 양자화(16비트→4~8비트로 낮춰 속도 향상, 단 성능 저하 트레이드오프) |
| 비용(Cost)        | 요금 폭증                                                          | 캐싱, 배치 처리                                                                        |
| 품질 변동         | 동일 입력에도 다른 출력(확률적 특성상 불가피)                      | temperature 고정, 로그 분석으로 원인 추적                                              |
| 안전(Safety)      | 유해·혐오 응답, 우회 질의(예: 소설 형식으로 위장한 위험 정보 요청) | 콘텐츠 필터링·사고 검증으로 가드레일 구축                                              |
| 장애(Reliability) | 서버 다운, 500 에러                                                | 주기적 헬스체크, 재시도(retry) 로직, 실패 시 폴백(대체 상용 LLM 호출 등)               |

강조 포인트: "레이턴시를 안다"고 말할 때 평균값만 보는 사람이 많지만, **P99·P90**(가장 오래 걸린 상위 1%·10% 사용자 기준)을 봐야 제대로 된 튜닝이 가능하다는 것 — 이것이 "실무를 안다"는 인상을 주는 디테일이라고 강조한다.

## 요구사항 → 기술 스펙 변환 실습 예시

발표자는 "자소서를 입력하면 AI가 문법과 내용을 다듬어 첨삭해줬으면 좋겠다"는 자연어 비즈니스 요구사항을, 아래처럼 기술 스펙으로 번역하는 과정을 보여준다.

- 입력 계약: 텍스트는 string, 컨텍스트 윈도우 문제로 최대 길이 5,000자 제한.
- 출력 스키마: 텍스트, diff 카운트 등.
- 시스템 제약(수치화): 레이턴시 P99 3,000ms 이하, 개인정보는 DB에 저장하지 않음.
- 예외 처리: 입력이 비면 400 에러 + 명확한 메시지.

이 변환 과정이 있어야 API 구현과 Docker 세팅이 가능해진다는 것이 핵심 메시지다.

## API 계약(Contract) 설계 — 클라이언트-서버 합의

예시 엔드포인트(`/invoke`): 클라이언트가 `{text, max_length, temperature(선택)}`를 JSON으로 보내면, 서버는 `{output, latency_ms, request_id, status(success/fail)}`을 응답한다. 핵심은 **디커플링(decoupling) 원칙** — 모델이 A에서 B로 바뀌든, 프롬프트 기반에서 RAG 기반으로 바뀌든, 심지어 에이전트로 바뀌든 **API 스키마만 유지되면 클라이언트 코드는 수정할 필요가 없다**. 이것이 서비스 지속성의 핵심이라고 강조한다.

## 구조화된 로그(Structured Logging) — 운영의 최소 단위

운영에서 가장 무서운 상황은 "사용자가 품질이 나쁘다고 불만을 제기했는데, 서버에 요청이 들어왔는지·어떤 응답을 줬는지조차 찾을 수 없는 것"이다. 이를 막기 위한 3요소:

1. **Request ID**: 추적의 열쇠. 이 하나로 어떤 입력이 있었고 어떤 에러가 났는지 찾을 수 있어야 한다.
2. **P99 레이턴시**: 평균이 아니라 꼬리(tail) 값을 봐야 튜닝이 가능.
3. **구조화된 JSON 로그**: 단순 print문 대신 JSON 형식으로 남겨야 ELK·CloudWatch 같은 로그 수집 툴에서 즉시 분석 가능. (필드 예시: timestamp, request_id, level, model, latency, status, input/output)

## 미니 실습 — 로컬 함수를 서비스로 승격시키는 3단계

강연 후반 실습에서 제시하는 최소 단위 패턴:

```
[Step 1: Define] 함수 골격 정의, 역할 명시 (텍스트 입력 → 텍스트 출력 계약)
        │
        ▼
[Step 2: Measure] 시작·끝 시간 측정 → 레이턴시 계산 (성능 지표 확보)
        │
        ▼
[Step 3: Log] 한 줄 로깅(가능하면 print보다 JSON) — 관측 가능성의 출발
        │
        ▼
이 3단계 완성 → FastAPI 라우터 안으로 옮기기만 하면 서비스로 전환 완료
```

완성 기준은 "대충 되면 된다"가 아니라 **성공/실패 시나리오를 먼저 정의**하는 것 — 예: 정상 입력 처리(반환 타입 string인가? 레이턴시가 로그로 찍히는가?) / 엣지 케이스(빈 입력에도 프로그램이 죽지 않고 명확한 에러 메시지를 반환하는가?).

## 실습 저장소 구조(후반부 데모)

발표자가 준비한 실습 저장소는 FastAPI + Ollama(로컬 LLM 백엔드) + Open WebUI(챗봇 프레임워크)를 결합해, GPU 없는 로컬 환경(맥북 프로 M3 예시)에서도 추론 서버를 구동하고 챗봇으로 확인할 수 있게 구성했다. 주요 엔드포인트:

- `GET /health` — 헬스체크
- `POST /infer` — 커스텀 추론
- `GET /v1/models` — OpenAI 호환 모델 목록
- `POST /v1/completions` — OpenAI 호환 채팅 완성(SSE 스트리밍 지원)

OpenAI 호환 엔드포인트를 쓰는 이유: 대부분의 오픈소스 챗봇 프레임워크·추론 라이브러리(vLLM 등)가 이 규격을 기준으로 서빙되며, OpenAI 파이썬 패키지의 `base_url`만 바꾸면 로컬 모델도 그대로 사용할 수 있어 호환성이 높다. 요구사항 명세서(requirements.md)에 헬스체크·추론 API·입력 검증·request ID·레이턴시·구조화 로그·Docker Compose 통합 실행까지 기능 요구사항으로 명문화해두고, `uv`로 의존성을 잠가(`uv lock`) 누구나 저장소를 클론하면 동일 환경에서 재현 가능하게 만든 것이 포인트다.

## 회계·감사 관점 메모

이 강연은 "LLM 서비스를 프로덕션급으로 만드는 법"을 다루지만, 핵심 원칙(계약 고정 + 구조화 로그 + Request ID 추적 + P99 기준 성능 관리)은 회계·감사 시스템에서 AI 기반 이상거래 탐지·문서 처리 파이프라인을 설계할 때도 동일하게 적용된다. 특히 "요청 단위로 무엇이 입력됐고 어떤 근거로 이 결론이 나왔는지 추적 가능해야 한다"는 원칙은 감사 증거(audit evidence) 요건과 직접 맞닿아 있다.
