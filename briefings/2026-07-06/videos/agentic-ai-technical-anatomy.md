# 영상 정리: Agentic AI의 기술적 해부 (RAG, A2A, MCP, MAS, ...)

> 원본: [Agentic AI의 기술적 해부 (RAG, A2A, MCP, MAS,...)](https://www.youtube.com/watch?v=aDukCWkPbeQ) — 한국어, 유튜브, 중급
> 연결된 브리핑: [2026-07-06.md](../../2026-07-06.md)
> 이 문서는 한국어 자막을 실제로 내려받아 확인하고 정리했다.

## 전체 구조 — 이 영상이 다루는 4개 층위

영상은 에이전틱 AI를 "두뇌(LLM) → 단일 에이전트 아키텍처 → 다중 에이전트 협업 → 에이전트 간 통신 프로토콜"의 4단계로 쌓아 올리며 설명한다.

```
[LLM: 예측 엔진]
      │  + 자율성·목표지향성·행동 능력 부여
      ▼
[단일 에이전트] = LLM + 도구 + 컨텍스트 + 계획 + 메모리
      │  혼자 풀기 힘든 문제는 분업
      ▼
[다중 에이전트 시스템(MAS)] = 여러 전문화된 에이전트의 협업 + 공유 메모리
      │  팀이 커지면 소통 규칙이 필요
      ▼
[통신 프로토콜: MCP, A2A 등] = 에이전트-도구, 에이전트-에이전트 표준 인터페이스
```

## LLM의 본질적 한계와 에이전틱 AI의 등장 배경

GPT-4 같은 LLM은 방대한 텍스트 학습으로 뛰어난 언어 능력을 갖췄지만, 본질적 한계는 **수동성**이다 — 명시적 지시 없이는 스스로 행동을 시작하지 않고, 대화가 끝나면 이전 상태를 기억하지 못하며, 오직 주어진 요청에만 반응한다. 에이전틱 AI는 이 한계를 극복하기 위해 LLM을 핵심 두뇌로 삼되 자율성·목표지향성·행동 능력을 부여하는 프레임워크를 결합한 것이다. 핵심 공식: **에이전틱 AI = LLM + 도구 + 컨텍스트 + 계획 + 메모리**.

## AI 에이전트 vs 에이전틱 AI — 규모의 차이

- **AI 에이전트**: 이메일 필터링, DB 쿼리처럼 좁고 잘 정의된 반복 작업을 맡는 "혼자 일하는 전문가".
- **에이전틱 AI**: 혼자서는 풀기 힘든 크고 복잡한 문제를 풀기 위해 여러 전문화된 에이전트가 협력하는 **시스템 자체**. 동적으로 작업을 나누고 공유 메모리를 활용하며, 개개인의 능력을 합친 것 이상의 "창발적 집단 지성"을 보인다.

## 단일 에이전트의 추론·계획 프레임워크

- **ReAct**: 생각(Thought)과 행동(Action)을 번갈아 하는 방식. "이걸 해야겠다" → 행동 결정 → 관찰(Observation) → 다음 생각. 환각(hallucination) 현상을 완화하지만, 행동 하나하나마다 LLM을 호출해야 해서 느리고 비용이 많이 든다.
- **Plan-and-Act (계층적 계획)**: 플래너(설계자) LLM이 큰 그림의 계획을 먼저 짜면, 익스큐터(실행자)가 그 계획대로 처리한다. 역할 분리는 컴퓨터 공학의 기본 원리인 "관심사의 분리(Separation of Concerns)"와 통한다.

## 메모리 — RAG 파이프라인

LLM의 컨텍스트 윈도우 한계를 극복하기 위해 외부 최신 정보를 실시간 참고하는 RAG(Retrieval-Augmented Generation, 검색 증강 생성) 기술을 사용한다.

```
외부 문서 ─▶ [청크 분할] ─▶ [임베딩: 벡터로 변환] ─▶ [벡터 DB 저장]
                                                            │
사용자 질문 ─▶ [질문도 벡터로 변환] ─▶ [코사인 유사도 등으로 벡터 DB 검색] ─┘
                                                            │
                                                            ▼
                              [원래 지식 + 검색된 정보 덩어리를 종합] ─▶ 최종 답변
```

이 파이프라인이 AI의 "장기 기억" 역할을 한다.

## 행동 — 함수 호출(Function Calling)

중요한 점은 **AI가 직접 함수를 실행하지 않는다**는 것. AI는 사람의 자연어 요청을 컴퓨터가 이해할 수 있는 명확한 명령서(JSON)로 바꾸는 "번역가" 역할만 한다. 실제 애플리케이션이 이 JSON을 받아 외부 실행을 담당하고, 결과를 다시 AI에 알려주면 AI가 최종 답변을 생성한다. 즉 AI 두뇌는 기억(벡터 DB)과 행동(외부 도구) 사이에서 전체 흐름을 지휘하는 "의미론적 라우터" 역할을 한다. 오케스트레이션 계층(LangChain, LangGraph, CrewAI 등)이 이 모든 구성요소를 묶어 생각·기억·행동의 순서를 조율하는 총 지휘자다.

## 다중 에이전트 협업 패턴 2가지

- **계층적 패턴(Hierarchical)**: 중앙 관리자 에이전트가 일을 나눠주고 결과를 합친다. MetaGPT가 대표 사례. 업무 흐름이 명확하지만 관리자가 막히면 팀 전체가 멈추는 위험(단일 장애점)이 있다.
- **P2P 분산 패턴(Peer-to-Peer)**: 팀장 없이 모든 에이전트가 동등하게 직접 협력. Microsoft AutoGen이 대표 사례. 유연하지만 의견 조율이 복잡해질 수 있다.

## 통신 프로토콜 2계열

- **컨텍스트 지향 프로토콜(MCP, Model Context Protocol)**: 에이전트와 도구 사이의 통신을 표준화. Google Drive·코딩 툴·이미지 처리 등 다양한 애플리케이션을 AI 에이전트에 손쉽게 통합할 수 있어 주목받고 있다.
- **에이전트간 프로토콜(A2A, Agent-to-Agent)**: 에이전트들이 서로 일을 부탁하고 진행 상황을 보고하며 협상할 때 사용. 시간이 오래 걸리는 작업을 위한 비동기 통신 지원이 핵심이다.

두 프로토콜의 발전은 AI 생태계가 개별 시스템을 넘어, 서로 다른 회사가 만든 AI끼리도 자유롭게 협력하는 "AI들의 인터넷" 시대로 나아가고 있음을 보여준다.

## 신뢰성·안전 이슈와 대응 방안

에이전틱 AI는 편리하지만 다음과 같은 복잡한 리스크를 동반한다.

- **스토캐스틱(확률적) 특성**: 매번 조금씩 다르게 판단해 행동을 100% 예측하기 어렵고, 이는 시스템 테스트·신뢰를 어렵게 만든다.
- **오류 전파**: 한 에이전트의 실수가 팀 전체로 퍼질 위험.
- **프롬프트 감염(Prompt Injection propagation)**: 악의적인 지시가 에이전트 간에 전파되는 공격.
- **목표 이탈(Objective Drift)**: 교묘하게 조작된 피드백으로 AI의 목표가 슬쩍 바뀌는 공격.
- **IAM(신원 및 접근 관리) 격차**: AI에게 안전하게 권한을 부여하는 문제.
- **설명 가능성 문제**: AI가 내린 결정의 이유를 설명하고 책임을 묻기 어려움.

이 모든 문제는 "AI의 최대 장점인 자율성"과 "안전하게 통제하려는 요구" 사이의 **자율성-제어 역설(Autonomy-Control Paradox)**로 요약된다. 대응 방안으로는 ① 민감한 작업에 인간 참여(HITL, Human-in-the-Loop) 최종 승인 필수화, ② 각 AI의 역할과 권한을 명확히 분리하는 모듈식 보안 아키텍처, ③ AI의 추론 과정까지 투명하게 기록하는 종합 감사 시스템, ④ "이 결과가 제대로 됐는지 검토해달라"고 요청하는 비평가(Critic) 에이전트를 팀에 두어 서로의 실수를 잡아내는 방법이 제시된다.

## 결론

에이전틱 AI의 발전은 단순히 더 똑똑한 두뇌를 만드는 문제가 아니라, 본질적으로 여러 부품(LLM·도구·메모리·계획·프로토콜)을 잘 조립하고 지휘하는 **시스템 공학**의 영역이다. RAG·함수 호출·ReAct·Plan-and-Act 같은 기술로 작동하고, 계층적·P2P 같은 패턴으로 협력하며, MCP·A2A 같은 프로토콜로 소통하는 방향으로 발전 중이지만, 자율성과 통제 사이의 딜레마를 얼마나 효과적으로 해결하느냐가 향후 성공의 관건이다.

## 회계·감사 관점 메모

이 영상에서 다룬 자율성-제어 역설과 HITL(인간 참여 최종 승인) 요구는 감사 실무에서 AI 에이전트를 활용한 지속적 감사(Continuous Auditing) 설계 시 그대로 적용된다. 특히 "비평가 에이전트"와 "종합 감사 시스템(추론 과정 기록)" 개념은 회계법인이 AI 감사 도구를 도입할 때 요구되는 감사증적(audit trail) 확보·이상 탐지 이중검증 구조와 직접 맞닿아 있다.
