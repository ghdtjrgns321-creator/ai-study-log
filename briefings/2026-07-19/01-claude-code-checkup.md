# 오늘의 AI 개념: 클로드 코드 체크업(Claude Code /checkup)

> 작성일: 2026-07-19 · 분류: claude-code

## 한 줄 정의

클로드 코드가 자신의 설정 파일과 확장 기능을 스스로 점검해서, 문제를 발견하면 사용자 승인을 받고 고쳐 주는 명령어다.

## 쉬운 설명

`/checkup`은 기존 `/doctor` 명령을 재구축한 것으로, 이번에는 진단에서 그치지 않고 실제 수정까지 수행한다. 여기서 CLAUDE.md는 프로젝트 규칙을 적어둔 설명서, MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)는 AI 에이전트가 외부 도구·데이터에 연결하는 표준 규격, 훅(hook)은 특정 시점에 자동 실행되는 스크립트를 뜻한다.

비유하자면 건강검진과 비슷하다. 병원에서 검사를 마치면 의사가 결과만 알려주는 게 아니라 "이 약을 드시겠습니까?"라고 물어본 뒤 처방까지 해 주는 셈이다.

기존 `/doctor`는 문제를 나열만 하고 사용자가 직접 고쳐야 했다. `/checkup`은 "먼저 보고하고, 승인을 받은 뒤에만 변경한다"는 원칙 아래 각 항목을 개별적으로 보여주고 확인받은 것만 적용한다는 점이 다르다.

## 동작 원리

`/checkup`은 다음 순서로 동작한다.

1. 현재 저장소의 `CLAUDE.md`, 설치된 스킬·MCP 서버·플러그인, 훅 설정, 최근 권한 거부 이력을 스캔한다.
2. 각 항목의 사용 빈도와 컨텍스트 비용(토큰 소모량)을 비교해 미사용 항목을 찾아낸다.
3. 로컬 `CLAUDE.md`와 저장소에 커밋된 `CLAUDE.md`를 비교해 중복 내용을 식별한다.
4. 느리게 실행되는 훅, 자주 거부되는 읽기 전용 명령, 구버전 클로드 코드 등 개선 가능한 항목을 목록화한다.
5. 발견한 문제를 항목별로 사용자에게 제시하고, 승인받은 항목만 개별적으로 적용한다.

## 비슷한 것과 비교

| 구분 | `/doctor`(구버전) | `/checkup`(현재) |
|------|------|------|
| 기능 | 문제 진단만 수행 | 진단 후 수정까지 수행 |
| 수정 방식 | 사용자가 직접 조치 | 항목별 승인 후 자동 적용 |
| 점검 범위 | 기본 환경 오류 위주 | 스킬·MCP·플러그인·훅·권한 이력까지 포괄 |

`/doctor`는 여전히 존재하며 `/checkup`은 그 별칭이므로, 둘 중 어느 쪽을 입력해도 동일하게 동작한다. 단순 오류 확인이 아니라 저장소 전체의 설정 위생을 관리하고 싶다면 `/checkup`을 정기적으로 실행하는 편이 낫다.

## 핵심 용어·논쟁

- **CLAUDE.md** — 클로드 코드에게 프로젝트 규칙·컨벤션을 알려주는 설명 파일. 저장소에 커밋해 팀과 공유할 수 있다.
- **MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)** — AI 에이전트가 외부 도구·데이터 소스에 표준화된 방식으로 연결하기 위한 규격.
- **훅(Hook)** — 도구 호출 전후 같은 특정 시점에 자동으로 실행되도록 등록해 둔 스크립트.
- **컨텍스트 비용(Context Cost)** — 스킬·MCP·플러그인 등을 로드할 때 소모되는 토큰(입력 분량) 비용.

`/checkup`은 승인 없이는 아무것도 바꾸지 않는다는 원칙을 내세우지만, 여러 항목을 한 번에 점검하다 보면 어떤 변경이 실제로 안전한지 사용자가 매번 판단하기 어렵다는 점은 여전히 실무에서 주의가 필요한 지점이다.

## 회계법인 AI 직무 연결 포인트

회계법인에서 클로드 코드 같은 에이전틱 코딩 도구를 감사·데이터 분석 자동화 스크립트 작성에 쓸 경우, 프로젝트마다 쌓인 설정·권한 규칙을 주기적으로 점검하지 않으면 예상치 못한 권한으로 파일을 수정하거나 민감한 재무 데이터에 접근하는 사고로 이어질 수 있다. `/checkup` 같은 자가 점검 기능은 이런 리스크를 정기적으로 줄여주는 통제 장치 역할을 한다.

여러 감사팀이 같은 저장소를 공유할 때 팀마다 다른 훅·플러그인이 누적되면 실행 속도가 느려지고 예측 불가능한 동작이 생기는데, 정기 점검 명령은 이런 협업 환경에서의 유지보수 부담을 낮춘다.
