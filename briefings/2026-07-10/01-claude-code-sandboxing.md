# 오늘의 AI 개념: Claude Code 샌드박싱(Sandboxing)

> 작성일: 2026-07-10 · 분류: claude-code

## 한 줄 정의

Claude Code 샌드박싱은 에이전트가 실행하는 셸 명령을 운영체제 수준에서 파일 시스템과 네트워크 접근 범위 안에 가둬, 매번 승인을 묻지 않고도 안전하게 자율적으로 작업하게 하는 기능이다.

## 쉬운 설명

Claude Code는 파일을 고치거나 명령을 실행할 때마다 사용자에게 승인을 구하는 구조였다. 안전하지만, 반복되는 승인 요청이 쌓이면 사용자는 내용을 제대로 보지 않고 습관적으로 승인 버튼을 누르게 되는 이른바 "승인 피로"가 생긴다.

샌드박싱은 이 문제를 다르게 푼다. 매 명령을 사람이 검토하는 대신, 애초에 명령이 닿을 수 있는 범위를 작업 디렉터리와 승인된 네트워크 도메인으로 제한해 버린다. 비유하면, 매번 문 앞에서 신분증을 확인하는 대신 애초에 들어갈 수 있는 방만 열어 두는 방식이다. 방을 벗어나는 행동 자체가 불가능하므로 문지기가 매번 나설 필요가 없다.

기존 권한 프롬프트(허용 목록, Accept Edits 모드 등)와는 층위가 다르다. 권한 규칙은 "이 명령을 실행해도 되는가"를 명령 실행 전에 판단하는 반면, 샌드박스는 일단 실행된 프로세스가 "실제로 어디까지 손댈 수 있는가"를 운영체제가 강제한다. 그래서 모델이 명령의 의도를 잘못 판단해도, 하위 프로세스가 무엇을 하든 경계 자체는 뚫리지 않는다.

## 동작 원리

샌드박스는 두 개의 경계로 구성된다.

파일 시스템 격리는 기본적으로 현재 작업 디렉터리와 세션 임시 디렉터리에만 쓰기를 허용하고, 나머지 파일은 읽기는 되지만 쓰기는 막는다. `~/.bashrc`나 `/bin/` 같은 시스템 경로는 명시적으로 허용하지 않는 한 수정할 수 없다.

네트워크 격리는 샌드박스 밖에서 도는 프록시 서버를 거친다. 처음 보는 도메인에 접속하려 하면 승인을 요청하고, 이후 같은 세션에서는 다시 묻지 않는다. `allowedDomains`로 미리 허용 목록을 정해 두면 이 프롬프트 자체를 없앨 수 있다.

macOS는 내장된 Seatbelt를, Linux·WSL2는 bubblewrap과 socat 패키지를 이용해 커널 수준에서 경계를 강제한다. 이 격리는 명령이 실행하는 모든 하위 프로세스에도 그대로 적용된다.

## 비슷한 것과 비교

| 구분 | 권한 규칙(Permission Rules) | 샌드박싱(/sandbox) |
|---|---|---|
| 통제 시점 | 명령 실행 전, 문자열 기준 판단 | 명령 실행 중, OS 수준 강제 |
| 적용 범위 | Bash·Read·Edit·WebFetch·MCP 등 모든 도구 | Bash 명령과 그 하위 프로세스만 |
| 우회 가능성 | 모델이 명령 의도를 잘못 분류하면 뚫릴 수 있음 | 프로세스가 실제로 무엇을 하든 OS가 차단 |
| 승인 UX | 승인 피로 발생 가능 | 자동 허용 모드로 프롬프트 대폭 감소 |

선택 기준은 명확하다. 어떤 도구를 쓸지 결정하는 문제라면 권한 규칙, 이미 실행이 승인된 명령이 실제로 무엇을 건드릴 수 있는지 제한하는 문제라면 샌드박싱이다. 둘은 상호 배타적이지 않고 함께 쓰는 게 기본값이다.

## 왜 지금 중요한가

Claude Code 공식 문서는 샌드박스 도입 배경으로 반복되는 권한 프롬프트로 인한 "승인 피로"를 명시적으로 언급하며, Bash 명령을 파일 시스템·네트워크 두 경계로 감싸는 구조를 설명한다. 2026년 시점 공식 문서 기준이며, 관리자를 위한 조직 차원 강제 설정까지 포함돼 있어 개인 도구를 넘어 기업 배포를 염두에 둔 것으로 보인다.

Anthropic 엔지니어링 블로그는 사내 사용 데이터를 근거로 샌드박싱 도입 후 권한 프롬프트가 84% 줄었다고 밝혔다. 자율성을 높이면서도 승인 절차를 없애지 않는 방향이라는 점에서, 에이전틱 코딩 도구의 보안 설계가 "매번 확인"에서 "경계 설계"로 옮겨가고 있음을 보여주는 사례다.

## 회계법인 AI 직무 연결 포인트

회계법인이 사내에 에이전틱 코딩 도구를 도입할 때 가장 먼저 걸리는 문제가 보안 통제다. 감사·재무 데이터를 다루는 환경에서는 에이전트가 실수로든 프롬프트 인젝션으로든 민감한 파일을 읽거나 외부로 전송할 위험을 어떻게 막을지가 도입 여부를 가르는 기준이 된다. 샌드박싱처럼 "권한을 매번 묻는 대신 애초에 손댈 수 있는 범위를 제한한다"는 설계 철학은 내부 통제 부서가 AI 도구의 안전성을 평가할 때 요구할 법한 통제 개념과 정확히 겹친다.

`sandbox.credentials` 설정으로 자격 증명 파일과 환경 변수를 별도로 차단하는 기능은, 감사 과정에서 클라이언트 시스템 접근 토큰이나 데이터베이스 자격 증명을 다루는 실무 환경에 그대로 적용할 수 있다. 관리자가 조직 차원에서 정책을 강제하고 개발자가 이를 임의로 넓히지 못하게 하는 구조도, IT 일반 통제(ITGC)에서 요구하는 직무 분리·변경 통제 원칙과 맞닿아 있다.

면접에서는 "AI 도구 도입 시 보안을 어떻게 확보하는가"라는 질문에, 권한 프롬프트 같은 절차적 통제뿐 아니라 OS 수준의 격리 경계라는 기술적 통제까지 층위를 나눠 설명할 수 있으면 실무 이해도를 보여줄 수 있다.

## 핵심 용어·논쟁

- 승인 피로(Approval Fatigue) — 반복되는 승인 요청에 지쳐 내용을 검토하지 않고 습관적으로 승인하게 되는 현상.
- 파일 시스템 격리 — 프로세스가 쓰기 가능한 경로를 작업 디렉터리 등으로 제한하는 통제.
- 네트워크 격리 — 프로세스가 접속 가능한 도메인을 승인 목록으로 제한하는 통제.
- bubblewrap — Linux에서 권한 없이도 네임스페이스 기반 격리를 구현하는 오픈소스 샌드박싱 도구.
- 도메인 프론팅(Domain Fronting) — 프록시가 호스트명만으로 허용 여부를 판단할 때, 이를 속여 차단되지 않은 통로로 우회 접속하는 기법.

공식 문서는 자체적으로 한계를 밝힌다. 기본 설정에서는 프록시가 TLS 트래픽 내용을 들여다보지 않기 때문에, 폭넓은 도메인을 허용하면 도메인 프론팅 같은 방식으로 데이터가 빠져나갈 여지가 남는다는 경고가 명시돼 있다. 강한 보안이 필요하면 TLS를 직접 종단하는 커스텀 프록시를 두라고 권고하는데, 이는 설치만 하면 끝나는 통제가 아니라 위협 모델에 맞춰 추가 설정이 필요한 절충적 통제임을 뜻한다.

## 자료 깊이 읽기

### Configure the sandboxed Bash tool (Claude Code 공식 문서) — 영어, 문서, 중급

Claude Code 샌드박스의 동작 원리를 다루는 공식 레퍼런스다. `/sandbox` 명령으로 모드(자동 허용/일반 권한)를 고르는 절차부터, 파일 시스템 쓰기 범위를 넓히는 `sandbox.filesystem.allowWrite`, 특정 자격 증명 파일과 환경 변수를 차단하는 `sandbox.credentials`, 조직 전체에 정책을 강제하는 managed settings까지 설정 예시와 함께 설명한다. macOS는 Seatbelt, Linux·WSL2는 bubblewrap·socat 조합으로 OS 수준 격리를 구현한다는 점과, 기본적으로 TLS 트래픽을 들여다보지 않아 도메인 프론팅에 취약할 수 있다는 한계까지 솔직하게 명시한 점이 특징이다. 실무에서 설정 파일을 직접 작성해야 할 때 참고할 1차 자료다.

### Making Claude Code more secure and autonomous with sandboxing (Anthropic 엔지니어링 블로그) — 영어, 블로그, 입문

Anthropic이 샌드박싱 기능을 공식 발표한 글이다. 필터시스템 격리와 네트워크 격리라는 두 경계를 왜 함께 둬야 하는지 설명하는데, 네트워크 격리가 없으면 침해된 에이전트가 SSH 키 같은 민감 파일을 외부로 빼돌릴 수 있고, 파일 시스템 격리가 없으면 침해된 에이전트가 시스템을 조작해 네트워크 접근권을 얻을 수 있다는 논리다. 사내 사용 데이터 기준 권한 프롬프트가 84% 줄었다는 수치를 근거로, 반복되는 승인 요청이 오히려 사용자의 주의력을 떨어뜨리는 "승인 피로" 문제를 해결하려 한다고 밝힌다.

## 자가 점검 질문

1. 권한 규칙(permission rules)과 샌드박싱은 통제 대상과 통제 시점에서 각각 어떻게 다른가.
2. 회계법인이 클라이언트 데이터에 접근하는 AI 에이전트를 도입할 때, 샌드박싱 개념을 어떤 내부 통제 요구사항에 대응시킬 수 있는가.
3. 샌드박스가 기본 설정에서 막지 못하는 위협은 무엇이며, 이를 보완하려면 어떤 추가 설정이 필요한가.
