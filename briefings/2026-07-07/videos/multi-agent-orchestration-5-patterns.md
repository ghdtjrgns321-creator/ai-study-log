> 연결된 브리핑: [02-multi-agent-orchestration.md](../02-multi-agent-orchestration.md)

## 왜 멀티 에이전트인가 (도입부 핵심 수치)

Gartner가 2024년 보고서에서 발표한 수치로, Agentic AI에 대한 기업 관심도가 1년 사이 **1,445%** 증가했다고 인용한다(강의 내 인용 수치이며, 원 보고서 대조는 별도 확인 필요). 단일 에이전트로 "경쟁사 분석 보고서 작성"처럼 복합 작업(웹 검색+데이터 정리+글쓰기)을 순차 처리시키면 컨텍스트 윈도우가 폭발하고 실패율이 오르며 디버깅이 어려워진다는 것이 멀티 에이전트가 필요한 이유다.

## 에이전트의 정의 — 3요소

강의는 "LLM에게 질문하고 답 받는 것"은 엄밀히 에이전트가 아니라고 못박는다. 진짜 에이전트는 다음 3요소의 반복 루프를 가진다.

```
[환경 인식 Perception] ──▶ [추론 Reasoning] ──▶ [행동 Action]
        ▲                                            │
        └────────────── 결과 관찰 ───────────────────┘
```

단순 LLM 호출은 입력→출력의 일방향 파이프인 반면, 에이전트는 결과를 보고 다음 행동을 결정하는 피드백 루프를 가진다는 점이 본질적 차이다.

## 멀티 에이전트 시스템의 4대 구성요소

| 구성요소         | 역할                                                                 |
| ---------------- | -------------------------------------------------------------------- |
| 오케스트레이터   | 전체 목표를 이해하고 서브 에이전트에 작업을 위임하는 지휘자          |
| 서브 에이전트    | 리서치·코드 작성·데이터 분석 등 특정 도메인에 최적화된 전문 에이전트 |
| 공유 상태·메모리 | 에이전트 간 정보 교환 저장소(단기·장기 메모리로 구분)                |
| 툴 레지스트리    | 에이전트가 쓸 수 있는 도구 목록과 스펙 카탈로그                      |

## 패턴 1 — ReAct (Reasoning + Acting)

2022년 Yao Shunyu 등의 논문에서 제안된 패턴. 핵심은 "행동하기 전에 반드시 생각하게 강제하는 것"이다.

```
Thought(현재 상황 분석·다음 행동 추론)
   │
   ▼
Action(툴 호출·외부 상호작용)
   │
   ▼
Observation(툴 결과 수신)
   │
   └──▶ 다시 Thought로 순환 ──▶ (충분하면) Finish → 최종 답변
```

예시: "2024년 한국 GDP는?" → Thought("최신 경제 데이터 필요, 웹 검색해야겠다") → Action(웹 검색 호출) → Observation(검색 결과 수신) → Thought("수치 확보, 정리 가능") → Finish. 각 Thought 내용은 프롬프트에 누적되어 컨텍스트로 유지된다.

**구현**: LangChain의 `create_react_agent(llm, tools)` 함수가 ReAct 프롬프트 템플릿을 자동 주입한다. `AgentExecutor(verbose=True)`로 실행하면 각 단계가 터미널에 출력되어 디버깅에 유리하며, `max_iterations`로 무한 루프를 방지하는 것이 실무 필수 설정이다.

**한계**: "지금 당장 무엇을 할지"만 생각하므로 복잡한 다단계 작업에서는 근시안적으로 행동하다가 전체 방향을 잃기 쉽다.

## 패턴 2 — Plan-and-Execute

ReAct의 한계(근시안적 판단)를 해결하기 위해 "계획과 실행을 분리하라"는 철학으로 등장. 플래너(전략가)가 목표를 서브태스크로 분해하고, 실행 에이전트(전술가)들이 계획에 따라 태스크를 수행한다.

```
사용자 요청: "경쟁사 3곳 분석 후 비교 보고서 작성"
        │
        ▼
[플래너] ── 태스크 그래프 생성
        │
   ┌────┼────┬─────────────┐
   ▼    ▼    ▼             │(의존성 없음 → 병렬 실행)
 태스크1 태스크2 태스크3     │
 (경쟁사A) (경쟁사B) (경쟁사C)
   └────┴────┴─────────────┘
             │ (모두 완료되어야 시작)
             ▼
        태스크4: 비교 분석·보고서 작성
             │
             ▼(실패/이상 시)
        플래너에게 피드백 → 재계획(동적 재계획)
```

**구현**: LangGraph의 `StateGraph`로 구현. `PlanExecuteState`(TypedDict)에 `plan`(태스크 목록)·`past_steps`(완료 기록)·`response`(최종 결과)를 상태로 관리. `planner_step`이 태스크 목록을 생성하고, `executor_step`이 plan의 첫 태스크를 꺼내 실행 후 `past_steps`에 추가. planner→executor→(조건부)executor 또는 종료로 이어지는 조건부 엣지가 재계획 로직의 핵심이다.

## 패턴 3 — Multi-Agent Collaboration (Supervisor-Worker)

핵심 아이디어는 "분업" — 기획자·개발자·QA처럼 각 에이전트가 고유한 페르소나와 책임을 갖는다.

```
                [Supervisor 에이전트]
             (요청 분석 → 라우팅 결정, 직접 실행은 안 함)
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
  [연구 에이전트]   [코딩 에이전트]   [리뷰 에이전트]
  요구사항 분석·     연구 결과 기반    코드 품질·보안·
  라이브러리 검색    코드 작성         컨벤션 점검
        └───────────────┴───────────────┘
                        │
                 결과 → Supervisor 반환
```

**구현**: 각 워커를 함수+고유 시스템 프롬프트로 정의 → 슈퍼바이저 체인 생성(라우팅 프롬프트+에이전트 목록) → `StateGraph`에 노드 등록, 슈퍼바이저의 라우팅 결과(`next` 필드)에 따라 조건부 엣지 연결. `next`가 `FINISH`이면 그래프 종료. 새 에이전트 추가는 노드 추가 + 멤버 리스트 갱신, 단 두 줄로 확장 가능하다는 것이 이 구조의 확장성 강점.

## 패턴 4 — Reflection (자기 개선)

Multi-Agent Collaboration이 "협업"이라면, Reflection은 "자기 개선"에 관한 패턴. 2023년 Reflexion 논문에서 공식화됐으며 코드 생성·에세이 작성·수학 문제 풀이에서 단일 패스 대비 성능 향상이 입증됐다.

```
1) Generator ── 초안 생성
        │
        ▼
2) Critic ── 약점·개선점 지적
        │
        ▼
3) Generator ── 비평 + 이전 시도 메모리 참고해 개선판 생성
        │
        ▼
4) 종료 조건 확인 ──아니오──▶ (2)로 순환
        │
       예(최대 반복 도달 or 품질 임계 통과)
        ▼
      최종 출력
```

**실무 설계 포인트**: 종료 조건 없이 돌리면 무한 반복에 빠지므로 최대 3~5회 반복 + Critic이 "APPROVED" 판정 시 조기 종료가 권장 패턴. `reflection_history` 리스트에 각 시도의 출력·비평을 누적해 다음 생성 시 참고하게 하면 같은 실수를 반복하지 않는다.

**적합/부적합**: 코드 생성·기술 문서·에세이 교정처럼 품질 기준이 명확한 태스크에 적합. 단순 검색·단답형 질의·실시간 응답 서비스에는 부적합(반복 호출로 응답 시간·비용 배증). Critic이 충분히 날카롭지 않으면 "다 좋습니다"만 반복하는 "아첨 루프" 위험도 있다.

## 패턴 5 — Tool Use

에이전트가 언어 능력만으로 해결 못하는 작업(실시간 검색, DB 조회, 코드 실행)을 외부 툴 연동으로 처리. 핵심 메커니즘은 Function Calling — LLM이 어떤 함수를 호출할지 JSON으로 결정하고, 실제 실행은 애플리케이션 레이어가 담당.

**툴 5대 유형**: ① 검색 툴(Tavily, Serper, Bing) ② 코드 실행기(Python REPL, E2B 샌드박스) ③ DB 툴(SQL, 벡터 DB) ④ 외부 API(날씨·금융·사내 시스템) ⑤ MCP(Model Context Protocol, Anthropic 제안 표준).

**구현**: LangChain `@tool` 데코레이터로 일반 함수를 툴로 변환(독스트링=툴 설명, 타입 힌트=JSON 스키마). `llm.bind_tools()`로 바인딩하면 대화 중 `tool_calls` 필드로 호출 의사 표현. `ToolNode`가 실제 함수를 실행하고 `ToolMessage`로 래핑해 반환. **안전한 실행 패턴**: 호출 결정 → 파라미터 유효성 검증 → 실행 → 성공 시 반환/실패 시 폴백(재시도·대체 툴·에러 메시지 전달).

## 패턴 비교표

| 패턴                      | 강점                                    | 약점                        |
| ------------------------- | --------------------------------------- | --------------------------- |
| ReAct                     | 복잡도 낮음, 구현 쉬움, 빠름            | 복잡한 멀티스텝 계획에 한계 |
| Plan-and-Execute          | 복잡한 태스크를 구조적으로 처리         | 초기 계획 비용 발생         |
| Multi-Agent Collaboration | 병렬 처리·전문화로 고성능               | 오케스트레이션 오버헤드     |
| Reflection                | 품질 중심 태스크에 탁월                 | 반복 루프로 지연시간 증가   |
| Tool Use                  | 외부 세계 연동의 핵심, 다른 패턴과 조합 | 단독으로는 완결된 패턴 아님 |

## 패턴 선택 결정 트리

1. 외부 데이터/툴이 필요한가? → 예: Tool Use를 기본 추가.
2. 단일 단계인가, 멀티스텝인가? → 단일: ReAct.
3. (멀티스텝이면) 사전 계획 수립이 가능한가? → 예: Plan-and-Execute.
4. 전문화된 역할 분리·병렬 처리가 필요한가? → 예: Multi-Agent Collaboration.
5. 산출물 품질을 반복 개선해야 하는가? → 예: Reflection.

## 하이브리드 조합 사례

- **ReAct + Reflection**: ReAct가 빠르게 초안을 만들고 Reflection이 품질을 검토·개선. 코드 생성·보고서 작성처럼 고품질 출력이 필요할 때 적합하나, LLM 호출이 2배로 늘어 비용·지연 증가.
- **Plan-and-Execute + Tool Use**: 계획 단계에서 툴 사용 순서를 결정하고 실행 단계에서 각 스텝에 맞는 툴을 호출. 리서치 자동화·데이터 파이프라인에 이상적.

패턴을 조합할수록 복잡도와 디버깅 난이도가 함께 오른다는 점을 주의사항으로 강조한다.

## 엔드투엔드 예제 — 리서치 자동화 에이전트 팀

사용자가 주제 입력 → 검색 에이전트가 자료 수집 → 요약 에이전트가 정리 → 비평 에이전트가 품질 검증(점수 7점 미만이면 요약 단계로 되돌아가는 조건부 분기, `should_revise` 함수) → 보고서 작성 에이전트가 최종 문서 출력. 검색 결과를 청크로 나눠 병렬 요약하는 부분은 Parallelization 패턴, 최종 보고서 생성은 Prompt Chaining 패턴을 적용한 것으로, 실무 시스템은 단일 패턴이 아니라 여러 패턴을 레이어처럼 조합해 완성된다는 점을 보여주는 예시다.

## 프로덕션 배포 전 점검 항목

1. **관찰 가능성**: LangSmith·OpenTelemetry로 각 에이전트의 입출력·실행 시간 추적.
2. **토큰 비용 관리**: 에이전트가 늘수록 토큰 소비가 기하급수적으로 증가 — 컨텍스트 압축·캐싱 필수.
3. **무한 루프 방지**: Evaluator-Optimizer(Reflection) 패턴에서 최대 반복 횟수 필수 설정.
4. **타임아웃**: 개별 에이전트 호출마다 타임아웃을 둬 단일 장애가 전체를 멈추지 않게 함.

## 실무에서 자주 빠지는 함정 3가지

1. **과도한 에이전트 분할**: 에이전트가 많을수록 좋다는 착각. 한 에이전트가 3단계 이상을 순차 처리할 수 있다면 분할하지 않는 것이 낫다.
2. **공유 상태 충돌**: 여러 에이전트가 같은 상태 필드를 동시에 수정하면 덮어쓰기 발생 — 에이전트별 전용 상태 필드를 할당할 것.
3. **컨텍스트 손실**: 에이전트 간 데이터 전달 시 배경 정보 누락 방지를 위해 명시적 핸드오프 객체를 사용할 것.

## 강의 최종 인사이트 3가지

1. 패턴은 도구일 뿐, 문제 정의가 먼저다(파이프라인=순차 처리, 라우터=조건 분기, 병렬화=독립 작업 분산, 오케스트레이터-서브에이전트=복잡한 계획 실행, 메모리 공유=상태 기반 협업).
2. 패턴을 조합하면 실제 문제의 90%를 해결할 수 있다.
3. 로깅·트레이싱(관찰 가능성) 없이는 멀티 에이전트 시스템을 운영할 수 없다.

추천 학습 로드맵: LangGraph 심화 → LangSmith/Ragas로 평가 프레임워크 학습 → AutoGen·CrewAI 등 고수준 프레임워크에서 오늘 배운 패턴이 어떻게 추상화되는지 비교.
