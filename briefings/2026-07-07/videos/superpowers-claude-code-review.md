# 영상 정리: Superpowers + Claude Code: Full Test & Honest Review

> 원본: [Superpowers + Claude Code: Full Test & Honest Review](https://www.youtube.com/watch?v=98e8lpOtaWc) — 영어, 유튜브, 약 10분
> 연결된 브리핑: [01-claude-code-agent-skills-plugin-ecosystem.md](../01-claude-code-agent-skills-plugin-ecosystem.md)
> 이 문서는 영상의 영어 자막을 실제로 내려받아 확인하고 정리했다.

## 실험 설계

제작자는 동일한 Shopify 앱을 **두 번** 만든다. 한 번은 Superpowers 플러그인을 켠 Claude Code로, 한 번은 순정(stock) Claude Code(Opus 4.6)로. 두 결과물을 비교해 "Superpowers가 실제로 결과물을 개선하는가"를 검증하는 것이 영상의 유일한 질문이다.

```
동일 프롬프트 + 동일 베이스 프로젝트
        │
        ├─▶ [Claude Code + Superpowers 14개 스킬] ──▶ 결과물 A (~25분 소요)
        │
        └─▶ [Claude Code 순정, Opus 4.6, Plan Mode만] ──▶ 결과물 B (~10분 소요)
                        │
                        ▼
              두 결과물을 기능·품질로 비교
```

## Superpowers 설치와 실행 구조

1. Claude Code 플러그인 마켓플레이스를 통해 GitHub 저장소가 안내하는 두 명령을 실행하고 재시작하면, `/skills` 목록에 Superpowers 소속 스킬 14개가 나타난다. 프로젝트 단위로 설치하는 것을 권장(전역 설치도 가능).
2. 첫 프롬프트 전에 `/using-superpowers`(영상 표현: "use superpowers" 슬래시 명령)를 수동으로 호출해 "스킬 프레임워크를 로드했다"는 상태로 만들어야 한다. 이 스킬의 역할은 모델이 항상 Superpowers 스킬 체계를 우선하도록 조종(steer)하는 것 — "레드 플래그" 사고 패턴 목록(예: "컨텍스트가 더 필요하다"는 생각이 들면 그건 사실 질문 전에 스킬 체크부터 해야 한다는 신호)을 모델에 주입한다. 워크플로가 손에 익으면 이 스킬 로딩 지침을 CLAUDE.md에 아예 박아두는 것을 제작자는 권장한다.

## Superpowers 파이프라인 — 브레인스토밍부터 검증까지

```
사용자 요청(고수준)
   │
   ▼
[brainstorming 스킬] ── 질문 다발로 요구사항 구체화
   │                     (예: 경쟁 상품 비교 기준을 뭘로 할지)
   ▼
2~3개 접근안 제시 + 추천안 1개
   │  (사용자가 이 지점에서 하나 선택)
   ▼
[writing-plans 스킬] ── 설계 문서 + 구현 계획(태스크 단위) 생성
   │
   ▼
[sub-agent-driven 스킬] ── 태스크별로 서브 에이전트 파견
   │        │
   │        ├─▶ 태스크 구현 (TDD: 테스트 먼저 작성 → red-green-refactor)
   │        ├─▶ [code-reviewer 스킬] 태스크별 코드 품질 리뷰
   │        └─▶ [spec-review] 계획 대비 구현 일치 여부 점검
   ▼
전체 태스크 완료
   │
   ▼
[verification-before-completion 스킬] ── 타입체크·전체 테스트·요구사항 재확인
   │
   ▼
최종 결과물 A
```

## 관찰된 장단점

**긍정적으로 평가한 지점**

- **브레인스토밍 스킬**: 제작자가 "가장 마음에 든 스킬"로 꼽는다. 단일 옵션을 던지는 대신 질문을 통해 사용자가 미처 생각 못한 대안들을 고려하게 만들고, 최종적으로 추천안 1개 + 대안 2개를 함께 제시하는 방식이 실질적으로 설계 품질을 높였다고 평가.
- **코드 리뷰 스킬**: 태스크 완료마다 자동으로 코드 품질 리뷰를 돌리는데, 실제로 구체적인 결함(예: "특정 조건에서 텍스트 길이 50자 이상 체크 없이 감점 처리")을 짚어내는 등 인상 깊었다고 평가.

**부정적/의문을 제기한 지점**

- **TDD 강제의 함정**: Superpowers는 테스트를 먼저 작성하는 구조를 강하게 밀어붙이는데, 제작자는 "LLM에게 실패하는 테스트를 던지고 통과시키라고 하면, 잘 설계된 해법보다 그 테스트만 통과하는 최소한의 땜질(hacky) 코드를 만드는 경향이 있다"는 근본적 우려를 제기한다. 테스트를 많이 준다고 해서 결과가 더 잘 설계된다는 보장은 없다는 것.
- **spec-review 단계의 실효성 의문**: 태스크마다 "계획대로 구현됐는지" 재확인하는 절차가 추가되는데, 이번 케이스에서는 "완벽히 일치한다"는 결과만 반복해 사실상 토큰만 소모한 오버헤드였다고 평가.
- **결과물 자체의 완성도**: Superpowers로 만든 버전은 상품 카탈로그를 스캔하는 기능은 동작했지만, 정작 요청한 핵심 기능인 "경쟁사 리스팅과의 비교"는 제대로 구현되지 않았다("competitor benchmarking is a big fail"). 브레인스토밍 단계에서 그렇게 자세히 질문했음에도 핵심 요구를 놓쳤다는 점을 지적한다.
- **소요 시간**: Superpowers 버전은 약 25분, 순정 버전은 약 10분 — 3배 가까운 시간 차이가 났다.

## 순정 Claude Code(Opus 4.6, Plan Mode)와의 비교

같은 프롬프트·같은 베이스 프로젝트를 순정 Claude Code의 내장 Plan Mode만으로 진행한 결과, 랜딩 페이지 구성이나 스캔 기능은 Superpowers 버전과 사실상 동일한 수준으로 나왔고, 오히려 소요 시간은 3분의 1이었다. 경쟁사 비교 기능의 완성도도 비슷한 수준(둘 다 부족)이었다.

## 결론 — 제작자의 최종 판단

1. Superpowers의 스킬 중 **일부는 유용하고 일부는 과하다**. 실제로 계속 쓰고 싶은 것은 `brainstorming`과 `code-reviewer` 두 개뿐이라고 밝힌다.
2. 더 큰 질문은 "이런 방법론 프레임워크(Superpowers뿐 아니라 spec-kit류 전체)가 여전히 가치가 있는가"이며, 답은 "그렇다, 그러나 그 가치는 매일 줄어들고 있다"는 것. 모델과 툴링(Claude Code 자체)이 새 버전마다 개선되면서, 이런 프레임워크가 강제하려는 규율(계획 수립, 이중 검증 등)을 네이티브 기능이 점점 대신하고 있기 때문이다. 예로 "6개월 전 Sonnet 4로 같은 테스트를 했다면 Superpowers가 훨씬 더 나은 결과를 냈을 것"이라 추정하지만, 현재의 Opus 4.6과 최신 Claude Code는 그 격차를 상당히 좁혔다고 평가한다.
3. 제작자가 더 중요하다고 보는 것은 프레임워크 자체보다 **"지금 쓰는 기술의 최신 문서와 스킬을 대화 컨텍스트 안에 정확히 넣어주는 것"**이라는 점을 강조하며 영상을 마친다.

## 회계·감사 관점 메모

이 영상은 코딩 방법론 비교이지만, 시사점은 감사 자동화 파이프라인 설계에도 적용된다. "TDD처럼 검증 절차를 촘촘히 넣을수록 안전하다"는 직관과 달리, 검증 단계가 늘어도 핵심 요구사항(여기서는 "경쟁사 비교")을 놓치면 무의미하다는 점 — 즉 **점검 항목의 개수보다 그 항목이 실제 감사 목적과 정합한지가 더 중요하다**는 교훈으로 옮겨볼 수 있다.
