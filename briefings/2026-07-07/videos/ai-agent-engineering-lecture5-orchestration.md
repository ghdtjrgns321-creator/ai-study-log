## 오케스트레이션의 정의

이 강의에서 오케스트레이션은 "단순히 도구 호출 순서만 정하는 것이 아니라, **컨텍스트를 어떻게 구성하고 계획하고 실행할 것인가를 결정하는 과정**"으로 정의된다. 사용자 요청을 파운데이션 모델·외부 도구·로컬 도구·데이터베이스로 조율해 처리하는 전체 과정을 가리킨다.

```
사용자 요청
    │
    ▼
[모델이 판단] ── 단순 작업? ──예──▶ 도구 1개 + 최소 컨텍스트로 즉시 처리
    │
   아니오(복잡한 워크플로우)
    │
    ▼
계획 수립 + 메모리 검색 + 매 스텝 컨텍스트 재조합 ──▶ 오케스트레이션 필요
```

## 에이전트 유형 6가지 (추론·계획·행동 방식에 따른 분류)

| 유형                           | 핵심 동작                                              | 적합한 상황                                                      |
| ------------------------------ | ------------------------------------------------------ | ---------------------------------------------------------------- |
| 반사형(Reflex)                 | 입력→행동 직접 매핑, 추론 없음                         | 키워드 정리, 단순 조회, 간단한 자동화                            |
| ReAct                          | 추론→행동→관찰 반복, 매 스텝 계획 갱신                 | 탐색형 워크플로우, 트러블슈팅, 동적 데이터 분석 (가장 널리 쓰임) |
| Plan-and-Execute               | 플래너/익스큐터 역할 분리                              | 디버깅이 쉬워야 하고 비용을 효율적으로 관리해야 할 때            |
| 쿼리 분해(Query Decomposition) | 복잡한 질문을 하위 질문으로 쪼개 각각 다른 도구로 해결 | 외부 지식 검색이 필요하고 답변을 근거에 기반시켜야 할 때         |
| 성찰형(Reflective/메타추론)    | 행동 후 자신의 결과가 올바른지 확인하고 수정           | 금융·의료 등 정확성이 중요한 사고 대응                           |
| 심층 리서치(Deep Research)     | 계획→쿼리 분해→ReAct 루프까지 결합한 종합형            | 학술 문헌 서베이, 기술 실사, 경쟁정보 분석 (비용·시간 소요 큼)   |

쿼리 분해 예시: "X와 Y 중 누가 더 오래 살았는가?" → "X는 몇 년 살았는가?"/"Y는 몇 년 살았는가?"로 분해 → 각각 검색 도구 호출 → 결과 비교해 결론 도출.

## 도구 선택 방식 3가지

1. **표준 선택(Standard Selection)**: 도구 사용과 동일한 방식. 구현이 단순하고 퓨샷 학습이 가능하지만, 도구가 많아지면 확장성이 저하된다. 도구 이름·설명을 간결하고 명확하게 다는 것이 중요. (예시: WolframAlpha API로 계산식을 처리하는 도구 선언·바인딩·실행 데모)
2. **시멘틱 도구 선택(Semantic Tool Selection)**: 도구가 많을 때 사용. 모든 도구 설명을 임베딩해 벡터 DB에 저장해두고, 쿼리가 들어오면 시멘틱 검색으로 가장 관련성 높은 도구를 가져온 뒤 파운데이션 모델이 최종 도구·파라미터를 결정.
3. **계층적 선택(Hierarchical Selection)**: 도구를 역할별 그룹으로 조직하고, 그룹 안에서 다시 선택. 비슷한 도구가 많을 때(예: 단순 계산기와 WolframAlpha가 공존) 정확도를 높이는 방식.

```
쿼리 ─▶ [도구 임베딩 인덱스에서 시멘틱 검색] ─▶ 상위 후보 도구 조회
                                                    │
                                                    ▼
                                    [파운데이션 모델이 관련 도구·파라미터 결정]
                                                    │
                                                    ▼
                                              도구 재호출·실행
```

## 도구 실행 토폴로지 — 단일/병렬/체인/그래프

- **단일 도구 실행**: 도구 하나, 파라미터 설정 → 실행 → 응답. 가장 흔한 기본형.
- **병렬 실행**: 서로 의존성 없는 여러 도구를 동시 호출. 강의 예시(고객 지원 티켓 처리)로 구조를 설명한다.

```
고객 문의(티켓 #1237)
        │
        ▼
   [오케스트레이션이 필요한 정보 판단]
        │
   ┌────┼─────────────────────┐
   ▼    ▼                     ▼
고객 상세정보   고객 주문 이력      고객 서비스 로그(이 티켓 내용)
(병렬, 서로 무관)                        │
                                          ▼
                              유사 티켓 검색 (이 로그를 알아야 검색 가능)
                                          │
                                          ▼
                                관련 지원 정책 조회 (유사 티켓을 알아야 조회 가능)
```

- 위 예시에서 "고객 상세정보·주문 이력·서비스 로그"는 서로 무관해 병렬 처리 가능하지만, "서비스 로그 → 유사 티켓 → 관련 정책"은 순차 의존 관계이므로 체인으로 묶인다. 즉 하나의 워크플로우 안에 병렬 구간과 체인 구간이 공존할 수 있다.
- **체인(Chain)**: 작업을 순차 실행, 각 단계가 이전 결과에 의존. LangChain의 LCEL(파이프 연산자, `prompt | llm`)로 구성하는 선형 워크플로우에 적합. 길이(최대 스텝 수)를 미리 정해두는 것이 좋다.
- **그래프(Graph)**: 조건부 분기·병합·비선형 워크플로우 모델링에 사용. LangGraph의 `StateGraph`로 구현하며, 조건부 엣지(conditional edge)로 분기를 설정하고 여러 경로가 결국 요약 노드로 수렴하도록 설계한다.

**그래프 예시(고객 지원 라우팅)**:

```
지원 요청 ─▶ [이슈 분류: 비용 문제? 기술 문제?] ──비용──▶ 인보이스 확인 ─┐
                                        │                                  ├─▶ [응답 요약(summarize_response)]
                                        └──기술──▶ 로그인 문제 처리 ──────┘
```

체인으로 충분하면 체인을, 분기가 많아지면 그래프를 선택하는 것이 강의의 실무 권장사항이다.

## 컨텍스트 엔지니어링 — 오케스트레이션의 연료

강의는 컨텍스트 엔지니어링을 "각 오케스트레이션 단계에서 필요한 정보와 지침을 올바르게 동적으로 조합하는 작업"으로 정의하며, 프롬프트 엔지니어링(무엇을 쓸지·어떤 결과를 받을지 결정)과 대비해 "어떤 정보를 근거로 답을 내게 할지 결정하는 것"이라고 구분한다.

컨텍스트 설정 원칙 4가지:

1. **관련성 우선**: 가장 유용한 정보만 선택적으로 검색.
2. **명료성**: 구조화된 포맷으로 예측 가능한 형태로 전달.
3. **토큰 최적화**: 긴 히스토리는 요약하고 불필요한 표현은 제거.
4. **동적 조립**: 매 스텝마다 현재 목표에 맞게 컨텍스트를 다시 조립.

## 오케스트레이션 설계 시 트레이드오프 체크리스트

- 시스템의 지연 시간과 정확도 사이의 트레이드오프를 신중히 검토.
- 시나리오에서 보통 필요한 액션 수를 파악해 복잡도를 결정.
- 이전 결과에 따라 계획이 얼마나 적응됐는지 평가.
- 다양한 계획 접근 방식을 비교·평가하고, 요구사항을 충족하는 가장 단순한 방식을 선택.
- 필요할 때만 점진적으로 복잡도를 키운다(과설계 방지).

## 이 영상과 위 "멀티 에이전트 오케스트레이션 패턴 5가지" 강의의 관계

두 영상은 같은 주제(오케스트레이션)를 다른 각도로 다룬다. GycnTkYEBYo 강의가 **패턴 중심**(ReAct·Plan-and-Execute·Multi-Agent Collaboration·Reflection·Tool Use)으로 "어떤 상황에 어떤 패턴을 쓸지"를 다뤘다면, 이 강의는 **설계 구성요소 중심**(에이전트 유형 분류, 도구 선택 방식, 도구 실행 토폴로지, 컨텍스트 엔지니어링)으로 "오케스트레이션을 이루는 각 부품을 어떻게 조립할지"를 다룬다. 두 영상을 함께 보면 패턴(무엇을 쓸지)과 구현 부품(어떻게 조립할지)이 서로 보완된다.
