# 오늘의 AI 개념: 클로드 코드 중첩 서브에이전트(Claude Code Nested Sub-agents)

> 작성일: 2026-07-12 · 분류: claude-code

## 한 줄 정의

Claude Code의 서브에이전트(위임받은 작업을 독립적으로 처리하는 보조 AI)가 이제 자기 자신의 서브에이전트를 최대 5단계까지 만들어낼 수 있게 된 기능이다.

## 쉬운 설명

기존에는 "메인 대화 → 서브에이전트"라는 1단계 위임만 가능했고, 서브에이전트가 하위 작업을 발견해도 자기 컨텍스트 안에서 직접 처리해야 했다.

v2.1.172(2026-06-10 출시)부터는 서브에이전트도 Agent 툴을 가질 수 있어 하위 서브에이전트를 만들 수 있고, 그 하위도 다시 하위를 만들 수 있다. 이 연쇄는 메인 대화를 0단계로 볼 때 5단계에서 강제로 끊긴다.

비유하면 팀장 밑의 팀원이 이제 자기 밑에 팀원을 두고 재위임할 수 있게 된 셈이지만, 5단계에 이르면 그 사람은 더 이상 조직도를 늘리지 못하고 직접 실무만 한다.

기존 "체이닝"(메인 대화가 A, B를 순서대로 호출)과 다른 점은 위임 결정을 서브에이전트 스스로 내린다는 것이다. 체이닝은 메인이 매번 다음 단계를 지시하지만, 중첩 구조는 중간 결과가 하위 트리 안에서 요약되어 최상위 서브에이전트로만 올라온다.

## 동작 원리

1. 메인 대화(깊이 0)가 서브에이전트 A(깊이 1)에 위임한다.
2. A의 `tools`에 `Agent`가 있으면 A는 서브에이전트 B(깊이 2)를 만들 수 있다.
3. 각 단계는 foreground·background와 무관하게 완전히 새 컨텍스트 창과 시스템 프롬프트로 시작한다(대화 전체를 이어받는 `fork`는 예외).
4. 깊이 5의 서브에이전트는 Agent 툴 자체를 받지 못해 더 생성할 수 없다. 이 상한은 고정값이며 설정 불가다.
5. 결과는 트리를 따라 위로만 요약되어, 메인 대화는 최상위 서브에이전트의 요약만 받는다.

## 구체 예시·사례

`code-reviewer`(깊이 1)가 PR 문제 10건을 찾아, 각 건마다 검증용 `verifier`(깊이 2)를 생성해 재현 여부를 확인시킨다. 그중 DB 마이그레이션 문제를 맡은 `verifier`는 직접 테스트 실행 서브에이전트(깊이 3)를 만들어 `pytest` 전체 스위트를 돌리고 실패 로그만 요약받는다. `pytest`의 방대한 출력은 메인 대화에 노출되지 않고 `code-reviewer`의 종합 요약 하나만 사용자에게 돌아온다. 특정 유형만 통제하려면 `tools: Agent(verifier)`로 허용 목록을 지정하거나, `permissions.deny`에 `"Agent(verifier)"`를 넣어 차단할 수 있다.

## 비슷한 것과 비교

| 구조 | 위임 결정 주체 | 깊이 제한 | 컨텍스트 시작 |
| --- | --- | --- | --- |
| 중첩 서브에이전트 | 각 단계 자신 | 고정 5단계 | 매 단계 새 창 |
| 서브에이전트 체이닝 | 메인 대화 | 제한 없음 | 매 호출 새 창 |
| 포크(fork) | 사용자·메인 대화 | 재포크 불가 | 부모 대화 전체 상속 |
| Codex 매니저-워커형 | 사용자가 직접 설정 | 사용자 설정값 | 도구마다 다름 |

하위 작업이 스스로 또 쪼개질 만큼 복잡하면 중첩 서브에이전트, 단계별 개입이 필요하면 체이닝, 직전 맥락을 통째로 넘겨야 하면 포크를 쓴다.

## 왜 지금 중요한가

Claude Code 공식 문서는 "Spawn nested subagents" 절에서 v2.1.172부터 이 기능이 도입됐고, 깊이는 실행 방식과 무관하게 메인 대화 아래 단계 수로 계산되며 깊이 5는 Agent 툴을 받지 못한다고 규정한다. 영어판·한국어판 문서 모두 동일하게 반영되어 있다.

GitHub 이슈 #67240은 이 기능이 v2.1.172에 정식 반영됐음에도 일부 문서에 구버전 설명("서브에이전트는 서로를 생성할 수 없다")이 남아 있었다는 정정 요청이며, 이후 v2.1.187(백그라운드 깊이 고정), v2.1.193(트리 패널 표시)에서 계속 다듬어지고 있어 지금도 진화 중인 기능임을 보여준다.

## 회계법인 AI 직무 연결 포인트

감사 업무는 파트너-매니저-시니어-주니어로 이어지는 계층적 위임 구조를 가지며, 이는 중첩 서브에이전트의 5단계 트리와 형태가 비슷하다. "전 계정과목 실증절차"를 맡은 에이전트가 계정과목별 서브에이전트를 만들고, 그 서브에이전트가 다시 표본 거래별 검증 에이전트를 만드는 워크플로를 그려볼 수 있다.

다만 토큰 비용이 트리의 각 분기마다 누적되므로, 계정과목과 표본이 많을수록 실행 비용이 급격히 늘어난다. 문서가 권장하듯 대부분의 유용한 체인이 2~3단계에서 끝난다는 점을 참고해 깊이를 제한하는 설계가 필요하다.

각 단계가 새 컨텍스트에서 시작하고 요약만 상위로 올라가는 구조는 "실무자의 상세 작업지"와 "매니저 검토 요약"을 구분 보관하는 방식과 닮았다. 다만 요약 과정에서 정보가 누락되지 않았는지 최하위 원본 로그를 재추적할 수 있는 체계가 중요한 통제 포인트가 된다.

## 핵심 용어·논쟁

- 서브에이전트 — 독립된 컨텍스트와 시스템 프롬프트로 특정 작업을 위임받아 처리하는 보조 에이전트.
- 깊이(Depth) — 메인 대화를 0으로 두었을 때 중첩 서브에이전트가 몇 단계 아래인지 세는 값.
- Agent 툴 — 서브에이전트가 하위 서브에이전트를 호출하는 도구로, 없으면 하위 생성이 불가능하다.
- 포크(Fork) — 부모 대화 전체를 이어받아 시작하는 서브에이전트로, 재포크는 불가능하다.

논쟁의 핵심은 고정 상한의 적절성이다. Claude Code는 깊이 5를 설정 불가능한 고정값으로 두었지만, Codex 같은 경쟁 도구는 사용자가 깊이·병렬 수를 직접 정하게 한다. 고정 상한 지지 측은 "깊어질수록 원래 지시에서 멀어져 품질이 떨어질 위험이 검증되지 않았다"는 점에서 안전한 기본값을 선호하고, 반대 측은 사용자의 비용·품질 저울질 자유를 제한한다고 본다.

## 자료 깊이 읽기

### Create custom subagents — 영어, 공식 문서, 중급
"Spawn nested subagents" 절에서 v2.1.172부터 서브에이전트가 자신의 서브에이전트를 생성할 수 있다고 명시한다. 깊이는 foreground·background 여부와 무관하게 메인 대화 아래 단계 수로 계산되며, 깊이 5는 Agent 툴을 받지 못해 고정 상한이라고 규정한다. v2.1.187부터는 백그라운드 서브에이전트를 재개해도 최초 생성 시점의 깊이가 유지된다. `tools`에서 `Agent` 생략이나 `disallowedTools`, `permissions.deny`의 `Agent(이름)` 형식으로 특정 유형을 차단하는 법을 제시하며, 포크는 재포크 불가하지만 다른 유형은 생성 가능하고 이 역시 깊이에 포함된다고 명시한다.

### v2.1.172 allows up to 5 levels of nesting (Issue #67240) — 영어, 이슈 트래커, 중급
이 기능이 v2.1.172 정식 출시임을 확인해주는 이슈다. 초기 문서 일부에 "서브에이전트는 서로를 생성할 수 없다"는 구버전 설명이 남아 신규 기능과 충돌했음을 지적하며 정정을 요청한 내용이 핵심이다. 기능이 버그나 소문이 아니라 공식 릴리스이고, 문서 갱신이 출시보다 다소 늦었다는 정황을 보여준다.
