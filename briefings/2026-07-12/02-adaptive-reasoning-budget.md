# 오늘의 AI 개념: 적응적 추론과 추론 예산(Adaptive Reasoning / Test-Time Compute Budget)

> 작성일: 2026-07-12 · 분류: ai-concept

## 한 줄 정의

AI 모델이 질문의 난이도에 맞춰 답을 도출하기 전 "생각"에 쓰는 연산량(토큰 수)을 스스로 늘리거나 줄이는 방식이다.

## 쉬운 설명

기존의 추론 모델 다수는 질문이 쉽든 어렵든 비슷한 양의 내부 사고 과정을 거친다. 그 결과 "2+2는?" 같은 간단한 질문에도 장문의 풀이를 늘어놓아 낭비하거나(overthinking), 복잡한 문제를 충분히 검토하지 않고 성급히 답하는(underthinking) 비효율이 함께 발생한다.

이는 신입 회계사가 단순 전표 확인에도 감사보고서 수준의 절차를 매번 거치거나, 반대로 복잡한 우발부채 판단을 첫인상만으로 끝내버리는 상황에 비유할 수 있다. 숙련된 실무자는 사안의 난이도를 먼저 가늠한 뒤 검토 강도를 조절하는데, 적응적 추론은 이 판단을 모델 스스로 하도록 만든다.

즉 적응적 추론은 입력 질문의 난이도나 모델 자신의 확신도를 근거로, 답변 전 사고 단계에 쓸 토큰 예산을 그때그때 다르게 책정한다. 쉬운 질문에는 최소한만 생각하고, 복잡한 문제에는 예산을 늘려 더 깊이 사고한다.

이는 개발자가 고정된 예산 값을 수동으로 지정하는 "제어형(controllable)" 방식과는 다르다. 최근 연구는 이를 L1-controllability(고정 예산 하의 수동 제어)와 L2-adaptiveness(입력별 자동 조절)로 구분하며, 적응적 추론은 후자에 해당한다.

## 동작 원리

1. 사용자 질문이 모델에 입력된다.
2. 모델(또는 별도 예측 모듈)이 질문의 난이도·복잡도를 추정한다.
3. 추정된 난이도에 따라 이번 응답에 쓸 사고 토큰 예산이 결정된다 — 쉬우면 적게, 어려우면 많이.
4. 정해진 예산 범위 안에서 단계별 추론(chain-of-thought)을 생성한다.
5. 예산 소진이나 충분한 확신 도달 시 사고를 종료하고 최종 답변을 만든다.

SelfBudgeter 같은 학습 기반 기법은 이 과정을 더 정교화한다. 모델이 문제를 보고 필요한 토큰 예산을 스스로 예측하도록 훈련(cold-start)한 뒤, 강화학습(budget-guided GRPO)으로 "예측한 예산을 지키면서도 정확도를 유지"하도록 다듬는다.

## 구체 예시·사례

한 회계법인 AI팀이 Claude API로 매입전표 자동 분류 파이프라인을 구축한다고 가정하자. 일반 전표 10,000건에는 `thinking.budget_tokens`를 문서상 최솟값인 1,024토큰으로 고정해 빠르고 저렴하게 계정과목만 분류한다. 반면 규칙 기반 필터에 걸린 예외 거래 300건에는 같은 모델 호출에서 `budget_tokens`를 10,000토큰 이상으로 올려, 거래 패턴과 과거 유사 사례를 여러 단계로 대조하며 정밀 분석하게 한다.

```python
thinking={"type": "enabled", "budget_tokens": 10000}
```

최신 모델(Opus 4.7 이후, Sonnet 5 등)에서는 이 수동 지정 대신 `thinking.type: "adaptive"`와 `effort` 파라미터로 전환되고 있다. 개발자가 토큰 수를 직접 정하지 않아도 모델이 요청의 복잡도를 보고 사고 깊이를 자동으로 결정한다.

## 비슷한 것과 비교

| 구분 | 고정 예산 제어(L1) | 적응적 추론(L2) | Best-of-N/자기일관성 샘플링 |
|---|---|---|---|
| 예산 결정 주체 | 개발자가 수동 지정 | 모델이 난이도 보고 자동 결정 | 고정 — 여러 번 생성 후 선택 |
| 대표 파라미터 | `budget_tokens`, `reasoning_effort=high` 고정 | `effort`(adaptive), SelfBudgeter | N(샘플 수) |
| 장점 | 비용 예측이 쉬움 | 평균 비용·지연 절감, 정확도 유지 | 정확도 향상 폭이 큼 |
| 단점 | 쉬운 문제도 과도 소비 가능 | 난이도 추정이 틀리면 손해 | N배 비용, 근본적 비효율 개선 아님 |

비용을 예측 가능하게 관리해야 하면 고정 예산 제어를, 문제 난이도 편차가 크고 평균 비용 절감이 중요하면 적응적 추론을 선택하는 것이 합리적이다.

## 왜 지금 중요한가

Anthropic 공식 문서는 Opus 4.5·Haiku 4.5까지 쓰이던 수동 `budget_tokens` 방식이 Opus 4.6/Sonnet 4.6에서 지원 중단 예고되고, Opus 4.7 이후 모델에서는 아예 지원하지 않으며 `effort` 기반 adaptive thinking으로 대체됐다고 안내한다. ([Extended thinking - Claude Platform Docs](https://platform.claude.com/docs/en/build-with-claude/extended-thinking))

OpenAI API 문서는 reasoning 모델 계열에서 `reasoning_effort`를 none/low/medium/high 등 여러 단계로 제공해, 지연시간·비용·정확도의 트레이드오프를 개발자가 상황별로 선택하도록 안내한다. ([Reasoning models | OpenAI API](https://developers.openai.com/api/docs/guides/reasoning))

국내 IT매체 전자신문은 2026년 2월 기사에서 토큰 예산이 제한된 조건에서 사고 방식에 따라 정확도가 달라지는 실험 결과를 다루며, "생각을 줄이면 오히려 똑똑해지는" 역설적 현상이 실무적으로 관찰되고 있다고 전했다. ([AI가 "생각을 줄이면" 오히려 더 똑똑해진다? 토큰 예산과 LLM 추론의 역설](https://www.etnews.com/20260224000361))

## 회계법인 AI 직무 연결 포인트

감사·재무 데이터는 양이 방대하지만 위험도는 균일하지 않다. 대량의 정형 전표나 계정 분류처럼 난이도가 낮은 작업은 저비용 모드(짧은 사고 예산)로 빠르게 처리하고, 부정 징후가 의심되는 소수의 이상거래는 고연산 모드로 정밀 분석하는 이중 구조를 설계하면 전체 AI 운영 비용을 크게 낮출 수 있다.

이는 기존 감사의 "위험 기반 접근(risk-based approach)"과 같은 논리다. 위험이 낮은 항목에는 최소 절차를, 높은 항목에는 심화 절차를 적용하는 감사 원칙이 AI 연산 자원 배분에도 그대로 적용된다.

면접에서는 어떤 기준으로 저비용/고비용 모드를 나눌지(거래 규모, 이례성 점수, 과거 오류 이력 등), 모델이 난이도를 잘못 판단해 중요한 이상거래를 저비용 모드로 흘려보낼 위험을 어떻게 통제할지에 대한 견해를 준비해두면 좋다.

## 핵심 용어·논쟁

- L1-controllability — 개발자가 고정된 연산 예산을 수동으로 지정해 통제하는 방식.
- L2-adaptiveness — 모델이 입력 난이도나 확신도를 근거로 연산 예산을 스스로 조절하는 방식.
- Overthinking/Underthinking — 쉬운 문제에 과도하게, 어려운 문제에 불충분하게 사고를 쓰는 비효율 현상.
- Thinking budget(사고 예산) — 최종 답변 전 내부 추론에 허용된 최대 토큰 수.
- Reasoning effort — 추론 깊이를 조절하는 단계형 파라미터(예: low/medium/high).

적응적 추론의 최대 쟁점은 예측 신뢰성이다. 모델이 스스로 매긴 난이도 판단이 틀리면 쉬운 문제에 과다 지출하거나, 실제로는 어려운 문제를 쉽다고 오판해 충분히 검토하지 않을 위험이 있다. 이 때문에 비용 예측이 중요한 조직에서는 여전히 고정 예산 제어를 선호하는 경우가 있고, 평균 비용 절감이 더 중요한 조직은 적응형으로 전환하는 추세라 두 방식이 당분간 공존할 것으로 보인다.

## 자료 깊이 읽기

### Reasoning on a Budget: A Survey of Adaptive and Controllable Test-Time Compute in LLMs — 영어, 논문(서베이), 고급
2025년 7월 공개된 서베이 논문으로, 현재의 추론 모델들이 문제 난이도와 무관하게 고정된 추론 연산량을 쓰는 탓에 쉬운 문제는 overthinking, 어려운 문제는 underthinking하는 비효율을 지적한다. 효율적 test-time compute 기법들을 L1-controllability(고정 예산 하 통제)와 L2-adaptiveness(입력 난이도·확신도 기반 동적 조절)라는 이단계 분류체계로 정리한 것이 핵심 기여다. 여러 상용 LLM을 다양한 데이터셋으로 벤치마크해 성능-토큰 사용량 트레이드오프를 정량 제시한다.

### SelfBudgeter: Adaptive Token Allocation for Efficient LLM Reasoning — 영어, 논문, 고급
베이징대 연구진이 제안한 자기적응형 추론 기법으로, 모델이 질문을 보고 필요한 토큰 예산을 먼저 스스로 추정한 뒤 그 예산 안에서 추론하도록 훈련한다. cold-start 단계에서 예산 예측을 표준 형식으로 학습시키고, 이어지는 budget-guided GRPO 강화학습 단계에서 난이도에 맞게 예산을 계획·준수하도록 다듬는다. 수학 추론 과제에서 평균 응답 길이를 61% 압축하면서도 정확도는 유지했다고 보고한다.

### Extended thinking — Claude Platform Docs — 영어, 공식 문서, 중급
Anthropic 공식 API 문서로, `budget_tokens`와 `effort` 두 파라미터를 비교한다. `budget_tokens`는 Opus 4.5·Haiku 4.5 등에서 지원되는 수동 지정 방식(최소 1,024토큰, `max_tokens`보다 작아야 함)이며, Opus 4.6/Sonnet 4.6부터 지원 중단이 예고되고 Opus 4.7 이후 모델에서는 제거되어 400 에러를 반환한다. 대신 `thinking.type: "adaptive"`와 `effort`를 쓰는 adaptive thinking이 기본값이 되어, 값을 지정하지 않아도 모델이 복잡도에 따라 사고 깊이를 자동 결정한다.
