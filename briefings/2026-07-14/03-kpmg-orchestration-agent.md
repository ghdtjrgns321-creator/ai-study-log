# 오늘의 AI 개념: KPMG 감사 오케스트레이션 에이전트(Orchestration Agent)

> 작성일: 2026-07-14 (2026-07-24 심화 개편) · 분류: audit

## 한 줄 정의

KPMG가 2026년 여름부터 시범 도입하는, 사람 대신 여러 개의 하위 AI 에이전트에게 감사 작업을 나눠 맡기고 지휘하는 상위 AI다.

## 쉬운 설명

오케스트라 지휘자가 각 악기 연주자에게 언제 무엇을 연주할지 지시하듯, 오케스트레이션 에이전트는 어떤 하위 에이전트가 어떤 감사 절차(자산 실사, 매출 프로세스 테스트)를 맡을지 스스로 판단하고 순서를 배정한다. 사람 감사인이 여러 도구를 하나씩 직접 실행하던 기존 방식과 달리, 이번에는 AI가 스스로 작업을 나눠 맡기는 역할까지 수행한다는 점이 다르다.

기존에 KPMG가 써오던 스마트 감사 플랫폼 '클라라(Clara)'는 생성형 AI 기능이 접목되어 문서 검토나 위험 식별을 돕는 도구였다면, 이번 오케스트레이션 에이전트는 그 도구들 위에서 "무엇을, 어떤 순서로, 누구에게" 시킬지를 결정하는 관리자 역할을 맡는다는 점에서 한 단계 위의 자동화다.

## 동작 원리

작업 흐름은 다음 다섯 단계다.

1. 감사인이 목표를 설정한다(예: 특정 계정과목에 대한 실증절차).
2. 오케스트레이션 에이전트가 필요한 하위 에이전트(문서 검토, 이상거래 탐지, 자산 실사 확인 등 20개 이상)를 선정한다.
3. 각 하위 에이전트가 병렬로 작업을 수행한다.
4. 오케스트레이션 에이전트가 결과를 취합하고 상호 검증한다.
5. 사람 감사인이 최종 판단을 내린다.

이 흐름을 받치는 인프라는 3개 층으로 쌓여 있다.

```
[거버넌스 층]  Microsoft Agent 365
  — 전사 에이전트의 배포·모니터링·보안·권한 관리
        ↓
[플랫폼 층]   KPMG Workbench (Azure AI Foundry 기반)
  — 에이전트 간(agent-to-agent) 상호통신, 멀티 LLM 지원,
    데이터 주권(어느 국가에 저장·처리할지) 제어
        ↓
[업무 층]     KPMG Clara(감사) · Digital Gateway(세무) · Velocity(자문)
  — 오케스트레이션 에이전트 + 하위 실무 에이전트들이 실제 절차 수행
```

KPMG가 2025년 6월 공개한 Workbench는 출시 시점에 50개 에이전트가 가동 중이고 약 1,000개가 개발 중이라고 밝힌 다중 에이전트 플랫폼으로, 특정 LLM에 종속되지 않게 여러 모델과 붙도록 설계됐다. 그 위의 거버넌스 층인 Agent 365는 마이크로소프트가 제공하는 에이전트 관리 계층으로, 2026년 6월 KPMG 전 세계 임직원 27만 6천여 명 규모로 확대 배포가 발표됐다. 오케스트레이션 에이전트는 이 스택의 업무 층에서 하위 에이전트를 지휘하는 존재다.

## 구체 예시·사례

첫 파일럿은 회사 창고나 설비 같은 물리적 자산 가치를 확인하는 실사 업무와, 기업의 매출 프로세스를 테스트하는 업무에 적용될 예정이다. 회계업계 매체 Going Concern이 KPMG 발표 자료를 토대로 정리한 로드맵은 구체적이다. 2026년 여름 파일럿 → 2027년 일부 테스트 항목에 에이전트 전면 배치 → 2029년 무렵 급여·매출 계약 검토 같은 일상적(routine) 테스트 전체를 에이전트가 수행. 주목할 부분은 KPMG의 역할 구분표에서 "일상 테스트" 항목만 사람 관여가 0으로 계획되어 있고, 위험 평가·데이터 수집·AI 작업 결과 검토는 계속 사람 감사인이 맡는다는 점이다. 즉 "감사인 대체"가 아니라 "테스트 실행의 위임 + 검토 책임의 유지"가 공식 설계다.

신뢰성 장치도 함께 공개됐다. KPMG는 AI 관리체계 국제표준 ISO/IEC 42001 인증(BSI 발급)을 받았고, Workbench의 모든 에이전트에 자체 'Trusted AI' 검증 표식을 부여한다고 밝혔다. 에이전트가 감사 절차를 수행하려면 그 에이전트 자체가 먼저 통제 대상이 된다는 구조다.

## 비슷한 것과 비교

혼동하기 쉬운 KPMG·마이크로소프트의 구성 요소를 층위로 구분하면 다음과 같다.

| 구분                    | 무엇인가                                   | 역할 층위   | 비유                |
| ----------------------- | ------------------------------------------ | ----------- | ------------------- |
| KPMG Clara              | 스마트 감사 플랫폼(생성형 AI 접목)         | 업무 도구   | 감사인의 작업대     |
| 오케스트레이션 에이전트 | 하위 에이전트 선정·배분·취합 AI            | 작업 관리자 | 현장 매니저         |
| KPMG Workbench          | Azure AI Foundry 기반 다중 에이전트 플랫폼 | 전사 플랫폼 | 에이전트들의 사무실 |
| Microsoft Agent 365     | 에이전트 배포·모니터링·보안 계층           | 거버넌스    | 인사·보안팀         |

선택 기준이라기보다 포함 관계다. Clara 안의 감사 에이전트들을 오케스트레이션 에이전트가 지휘하고, 그 전체가 Workbench 위에서 돌며, Agent 365가 감독한다. "에이전트를 쓰는가"보다 "에이전트를 관리하는 층이 있는가"가 이 구조의 차별점이다.

## 왜 지금 중요한가

KPMG는 2026년 여름부터 오케스트레이션 에이전트를 감사 실무에 시범 적용한다고 밝혔으며, 이는 KPMG가 마이크로소프트와 맺은 5년간 20억 달러 규모의 AI 투자·파트너십의 연장선에 있다. 2026년 6월에는 Agent 365와 Copilot을 KPMG 전 세계 직원 약 27만 6천 명 규모로 확장한다고 발표했다. 글로벌 감사부문 대표 Scott Flynn은 이 배포가 "실시간 분석과 더 이른 위험 식별"을 위한 것이라고 설명했다.

## 회계법인 AI 직무 연결 포인트

반복적인 실증절차(자산 실사 확인, 거래 테스트)가 자동화되면 감사인은 고위험 판단이 필요한 영역에 더 집중할 수 있게 된다. 다만 KPMG의 역할 구분표가 보여주듯, 사람에게 남는 핵심 업무가 "AI 작업 결과 검토"라면 감사인의 역량 요건 자체가 바뀐다. 절차를 직접 수행해본 경험 없이 AI 산출물의 적정성을 판단할 수 있는가 — 신입 감사인의 훈련 경로가 끊긴다는 문제와, 해외 위탁(offshore) 인력이 맡던 일상 테스트가 가장 먼저 대체된다는 고용 구조 문제가 Going Concern 기사에서도 지적됐다.

감사품질 관점에서는 오케스트레이션 계층의 로그가 사실상 감사조서(workpaper)의 일부가 된다. 어떤 하위 에이전트가 어떤 근거로 선정됐고, 병렬 수행 결과가 어떻게 취합·상호검증됐는지가 기록되지 않으면, 감리(감독기관의 품질 검사)에서 "이 감사증거는 누가 어떤 절차로 만들었는가"에 답할 수 없다. KPMG가 ISO/IEC 42001 인증과 에이전트별 Trusted AI 표식을 앞세우는 것도, 에이전트 자체를 통제 대상으로 문서화해 이 질문에 대비하려는 포석으로 읽을 수 있다.

국내 도입 관점에서는, 글로벌 KPMG가 "거버넌스 층(Agent 365)과 플랫폼 층(Workbench)을 먼저 깔고 그 위에 감사 에이전트를 올리는" 순서를 택했다는 점이 참고가 된다. 에이전트를 먼저 만들고 관리 체계를 나중에 붙이는 접근과 비교했을 때, 어느 쪽이 감독기관 대응에 유리한지가 국내 법인들의 로드맵 설계 쟁점이 될 것이다.

## 핵심 용어·논쟁

- 오케스트레이션 에이전트(Orchestration Agent) — 여러 하위 에이전트의 작업을 배분·조율하는 상위 에이전트.
- 서브에이전트(Sub-agent) — 특정 업무만 담당하는 하위 단위 에이전트.
- Agent 365 — 마이크로소프트가 제공하는 에이전트 배포·모니터링·거버넌스 계층.
- KPMG Workbench — Azure AI Foundry 기반의 KPMG 다중 에이전트 플랫폼. 에이전트 간 상호통신과 멀티 LLM·데이터 주권 제어를 제공.
- ISO/IEC 42001 — 조직의 AI 관리체계(AIMS)를 제3자가 인증하는 국제표준. KPMG는 BSI로부터 인증을 받았다.

현재 쟁점은 "사람의 개별 검토 없이 여러 에이전트가 처리한 결과를, 감사의견의 근거로 얼마나 인정할 수 있는가"이다. 자동화 범위가 넓어질수록 최종 책임을 누가 지는지, 오류 발생 시 어느 단계에서 걸러졌어야 했는지를 규명하는 절차가 함께 요구된다. 2029년 "일상 테스트 사람 관여 0" 목표가 감사기준상 "충분하고 적합한 감사증거" 요건과 어떻게 조화될지는 아직 감독기관의 답이 없는 영역이다.

## 자료 깊이 읽기

### KPMG and Microsoft scale trusted, enterprise AI agents globally (Microsoft Source) — 영어, 공식 발표 자료, 중급
2026년 6월 9일자 마이크로소프트 공식 발표로, 본문을 직접 확인해 요약했다. KPMG 전 세계 27만 6천여 임직원에게 Microsoft 365 Copilot과 Agent 365를 배포하고, Azure AI Foundry 기반 KPMG Workbench가 감사(Clara)·세무·자문 전반의 에이전트를 중앙 거버넌스 아래 조율한다는 내용이다. 글로벌 최고디지털책임자 Lisa Heneghan은 "거버넌스·가시성·책임성의 강한 기반"을 전제 조건으로 꼽았고, 감사부문 대표 Scott Flynn은 Copilot·Agent 365 내장이 "실시간 분석, 더 이른 위험 식별, 더 깊은 인사이트"를 가능케 한다고 밝혔다. 에이전트 확산 발표에서 거버넌스 계층을 전면에 내세운 구성 자체가 회계법인다운 접근이다.

### KPMG launches KPMG Workbench (KPMG 공식 보도자료) — 영어, 공식 보도자료, 중급
2025년 6월 17일자 Workbench 출시 보도자료로, 본문을 직접 확인해 요약했다. 출시 시점 50개 에이전트 가동·약 1,000개 개발 중, Azure AI Foundry 기반 에이전트 간 상호통신, Clara(감사)·Digital Gateway(세무)·Velocity(자문) 등 고객 서비스 플랫폼의 공통 기반 역할, 고객이 데이터 저장·처리 위치를 통제하는 데이터 주권 기능이 골자다. 글로벌 AI 총괄 David Rowlands가 "고객은 한 공급자에 묶이기보다 멀티모델 플랫폼을 원한다"고 설명한 대목과, ISO/IEC 42001 인증·전 에이전트 Trusted AI 표식 부여가 신뢰성 장치로 제시된 대목이 핵심이다.

### KPMG Plans to Hand Routine Testing Off to AI (Going Concern) — 영어, 회계업계 전문 매체, 입문
KPMG의 발표 자료를 토대로 로드맵을 비판적으로 정리한 기사로, 본문을 직접 확인해 요약했다. 오케스트레이션 에이전트가 20개 이상의 하위 에이전트를 자율 관리하는 구조, 2026년 여름 파일럿 → 2027년 일부 테스트 전면 배치 → 2029년경 일상 테스트 완전 자동화 일정, 그리고 역할 구분표에서 일상 테스트만 사람 관여가 없고 위험 평가·데이터 수집·AI 결과 검토는 사람이 유지한다는 세부를 담았다. 자동화 실패 시 책임 소재와 해외 위탁 인력의 고용 영향을 짚는 비판적 관점이 공식 발표 자료를 보완한다.

## 자가 점검 질문

1. Clara·오케스트레이션 에이전트·Workbench·Agent 365는 각각 어느 층위의 구성 요소이며, 왜 KPMG는 거버넌스 층을 먼저 깔고 감사 에이전트를 올리는 순서를 택했을까?
2. 여러 하위 에이전트가 병렬로 처리한 감사 증거를 취합할 때, 상호 모순되는 결과가 나오면 어떻게 처리해야 하는가? 그 처리 과정의 로그는 감사조서에서 어떤 지위를 가져야 하는가?
3. "일상 테스트는 사람 관여 0, 검토는 사람 유지"라는 KPMG의 역할 설계는 신입 감사인 훈련 경로에 어떤 문제를 낳는가? 국내 감사기준·감리 환경에서는 어떤 추가 제약이 예상되는가?
