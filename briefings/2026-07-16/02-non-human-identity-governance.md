# 오늘의 AI 개념: 비인간 신원 관리(Non-Human Identity, NHI Governance)

> 작성일: 2026-07-16 (2026-07-24 심화 개편) · 분류: ai-concept

## 한 줄 정의

사람이 아니라 AI 에이전트나 프로그램이 시스템에 접속할 때도, 사람 직원과 마찬가지로 고유한 신원을 부여하고 권한을 관리해야 한다는 개념이다.

## 쉬운 설명

지금까지 기업의 접근 권한 관리는 "이 계정은 김 대리 것, 저 계정은 박 과장 것"처럼 사람 단위로 이뤄졌다. 그런데 AI 에이전트가 스스로 데이터베이스를 조회하고, 문서를 수정하고, 다른 시스템을 호출하기 시작하면서, 그 에이전트 자체에도 계정과 권한이 필요해졌다. 이것이 비인간 신원(NHI)이다.

비유하면 예전에는 건물 출입증을 사람에게만 발급했는데, 이제는 자율주행 청소 로봇에게도 출입증을 발급해야 하는 상황과 같다. 로봇이 어느 층까지 들어갈 수 있는지, 언제 출입했는지 기록이 남아야 사고가 났을 때 추적할 수 있다.

기존의 서비스 계정(service account)이나 API 키와 다른 점은, AI 에이전트는 스스로 판단해 여러 단계의 작업을 연쇄적으로 수행하고, 필요하면 하위 에이전트까지 만들어낸다는 것이다. 발급 후 몇 년씩 방치되는 정적 API 키 하나로는 "이 에이전트가 지금 무엇을 하려고 이 권한을 쓰는지"를 설명할 수 없어, 업계는 "장기 보관하는 비밀번호를 없애고, 필요한 순간에만 짧게 신원을 발급한다"는 방향으로 옮겨가고 있다.

## 동작 원리

NHI 관리의 기술적 뼈대는 두 가지다. **워크로드 신원(workload identity)**과 **적시(JIT, Just-in-Time) 자격증명**이다.

워크로드 신원의 사실상 표준은 오픈소스 표준 SPIFFE(스피피)와 그 구현체 SPIRE다. 핵심은 "비밀번호를 미리 심어두지 않고, 실행 환경 자체를 검증해 신원을 발급한다"는 것이다.

```
1. 등록      관리자가 "어떤 조건의 워크로드에 어떤 신원을 줄지" 정책 등록
             (예: 이 쿠버네티스 네임스페이스·이 컨테이너 이미지 = 감사보조에이전트)
2. 증명      SPIRE가 노드(서버)와 워크로드(프로세스)의 속성을 실제로 검사
             — 네임스페이스, 서비스 계정, 이미지 해시 등 (attestation)
3. 발급      검증 통과 시 SVID(짧은 수명의 암호학적 신원 증서) 발급
             — spiffe://회사도메인/audit-agent 형태의 ID + 단기 인증서
4. 사용      에이전트는 이 SVID로 다른 시스템과 상호 TLS 인증
5. 회전      인증서가 만료 전 자동 재발급 — 훔쳐가도 곧 무효가 된다
```

JIT 자격증명은 그 위에 얹히는 권한 계층이다. 상시 권한(standing privilege)을 0으로 두고, 에이전트가 작업을 요청하는 순간에만 그 작업 범위로 좁힌 자격증명을 수 분~수 시간짜리 유효기간(TTL)으로 발급하고, 끝나면 자동 회수한다. CSA(Cloud Security Alliance) 백서는 에이전트 신원의 수명주기를 발급(코드로 관리, 소유자·목적·만료일 메타데이터 필수) → 운영(세션 시작 시 한 번이 아니라 요청 단위로 권한 재평가) → 회전(수 시간~수 일 단위) → 폐기(유출 징후 시 수 분 내 회수)로 규정한다.

## 구체 예시·사례

감사 보조 AI 에이전트가 클라이언트 ERP 데이터베이스에서 매출채권 원장을 추출하는 상황을 따라가 보자.

1. 에이전트가 컨테이너로 기동되면, 같은 서버의 SPIRE 에이전트가 "이 프로세스는 승인된 이미지에서, 승인된 네임스페이스에서 떴는가"를 검사하고 SVID를 발급한다. 에이전트 코드 어디에도 비밀번호는 없다.
2. 에이전트는 SVID로 자격증명 브로커(시크릿 매니저)에 인증하고 "AR 원장 읽기" 권한을 요청한다.
3. 브로커는 정책을 확인한 뒤 **읽기 전용, 매출채권 테이블 한정, 유효기간 15분**짜리 DB 접속 계정을 즉석에서 만들어 건넨다.
4. 에이전트가 쿼리를 실행한다. DB 로그에는 사람 계정이 아니라 `spiffe://.../audit-agent`가 남고, "어느 작업(engagement)을 위해 발급된 자격증명인지" 메타데이터가 함께 기록된다.
5. 15분 뒤 계정은 자동 삭제된다. 만약 이 자격증명이 코드 저장소에 유출돼도 이미 죽은 계정이다.

정적 API 키 방식이었다면 "전체 DB 읽기·쓰기 권한의 키가 3년째 어딘가에 살아 있는" 상태와 비교해 보면, 유출 시 피해 범위(blast radius)의 차이가 극명하다.

## 비슷한 것과 비교

| 구분      | 사람 계정(Human Identity) | 기존 서비스 계정/API 키         | 비인간 신원(NHI, 에이전트)                 |
| --------- | ------------------------- | ------------------------------- | ------------------------------------------ |
| 수명 주기 | 입사~퇴사, 관리 체계 성숙 | 장기간 고정, 관리 소홀하기 쉬움 | 작업 단위로 생성·소멸, 수량 폭증           |
| 신원 증명 | 비밀번호+MFA              | 미리 심어둔 정적 시크릿         | 실행 환경 증명(attestation) 기반 단기 증서 |
| 권한 판단 | 사람이 요청 시 검토       | 발급 시 고정                    | JIT 발급·요청 단위 재평가                  |
| 감사 추적 | 로그인 기록 위주          | 키 사용 로그                    | 어떤 작업을, 왜 수행했는지까지 필요        |

사람 계정 관리 체계를 그대로 에이전트에 적용하면 규모와 속도를 감당하지 못하므로, 별도의 거버넌스 체계를 새로 설계해야 한다는 것이 선택 기준이다.

## 왜 지금 중요한가

CSA 백서에 따르면 비인간 신원은 이미 사람 계정의 평균 45배, 클라우드 네이티브 환경에서는 최대 144배에 달한다. 그런데 NHI 기반 공격을 막을 수 있다고 자신하는 조직은 15%뿐이고, AI 신원의 소유자(책임자)가 불분명하다는 조직이 51%다. 2025년 한 해 GitHub에 노출된 AI 관련 시크릿만 127만 건(전년 대비 81% 증가)으로, "먼저 도입하고 나중에 통제한다"는 관행이 만든 공백이 수치로 드러나고 있다.

## 회계법인 AI 직무 연결 포인트

ITGC(IT 일반통제) 접근통제 평가의 전통 절차는 "계정 목록을 뽑아 재직자 명단·권한 승인 문서와 대조"하는 것이다. 그런데 JIT 방식에서는 계정이 15분 만에 생겼다 사라지므로, 특정 시점의 계정 목록이라는 감사 증거 자체가 무의미해진다. 평가 대상이 "계정"에서 "발급 정책과 발급 로그"로 이동한다 — 즉 감사인은 "누가 이 권한을 갖고 있나"가 아니라 "어떤 조건에서 권한이 발급되도록 설계돼 있고, 실제 발급 이력이 정책과 일치하나"를 검증해야 한다.

감사 증거의 신뢰성 측면에서는, 데이터 추출을 에이전트가 수행했다면 그 추출물이 증거로 성립하려면 "어떤 신원이, 어떤 범위 권한으로, 어떤 데이터에 접근했는지"가 끊김 없이 연결돼야 한다. SVID처럼 암호학적으로 검증 가능한 신원 체계는 이 연결 고리를 제공하는 반면, 공유 API 키 환경에서는 "누가 한 일인지" 자체를 특정할 수 없어 증거력이 흔들린다.

회계법인 자신이 감사 자동화 에이전트를 도입할 때도 같은 구조가 적용된다. 클라이언트 데이터에 접근하는 에이전트의 자격증명이 상시 권한인지 JIT인지, 유출 시 몇 분 내 회수되는지는 법인의 정보보안 실사와 품질관리에서 점점 구체적인 점검 항목이 되고 있다.

## 핵심 용어·논쟁

- 비인간 신원(NHI, Non-Human Identity) — 사람이 아닌 소프트웨어·에이전트에 부여되는 고유 식별 단위
- SPIFFE/SPIRE — 워크로드 신원의 오픈소스 표준과 그 구현체. 실행 환경 증명 기반으로 단기 신원 증서를 발급
- SVID(SPIFFE Verifiable Identity Document) — 워크로드가 자기 신원을 암호학적으로 증명하는 단기 증서(X.509 또는 JWT 형식)
- JIT 자격증명(Just-in-Time credential) — 작업 요청 시점에만 최소 범위·단기 유효기간으로 발급하고 자동 회수하는 권한 방식
- 상시 권한 제로(Zero Standing Privilege) — 평상시 보유 권한을 0으로 두는 설계 원칙
- 논쟁: 도구 도입 속도가 거버넌스 구축 속도보다 훨씬 빠르다는 점 외에도, "그림자 AI"를 금지 일변도로 막으면 가시성만 잃는다는 지적이 있다. CSA는 금지 대신 "승인된 셀프서비스 경로를 더 쉽게 만들어 통제된 길이 최소 저항 경로가 되게 하라"고 권고한다.

## 자료 깊이 읽기

### The Non-Human Identity Governance Vacuum — 영어, CSA 백서, 고급
Cloud Security Alliance 백서로, AI 에이전트는 정적인 자격증명 보유자가 아니라 권한을 동적으로 획득하고 하위 에이전트를 생성하며 시스템을 연쇄 호출하는 "자율 행위자"라서, 기존 서비스 계정 관리와는 다른 범주로 다뤄야 한다고 주장한다. 거버넌스 3축으로 (1) 중앙 NHI 레지스트리와 소유자 지정 등 수명주기 관리, (2) JIT 발급과 자동 회수 기반의 상시 권한 제로, (3) SPIFFE/SPIRE 기반 암호학적 워크로드 증명을 제시한다. 에이전트 자격증명의 수명을 "개월"이 아니라 "수 시간~수 일"로, 유출 시 회수 목표를 "수 분"으로 못 박은 구체성이 실무 점검표로 쓰기 좋다. 45:1 신원 비율, 15% 방어 자신감 등 이 브리핑의 통계 출처다.

### SPIFFE Concepts — 영어, 공식 기술 문서, 중급
SPIFFE 공식 문서로, 워크로드 신원의 구성 요소를 정의한다. SPIFFE ID는 `spiffe://신뢰도메인/워크로드` 형태의 URI이고, SVID는 그 신원을 증명하는 문서로 X.509 인증서형(재사용 공격에 강해 권장)과 JWT 토큰형(중간에 L7 프록시가 있을 때 사용) 두 가지가 있다. 핵심 설계는 Workload API가 클라우드 메타데이터 서비스처럼 "사전 배포된 시크릿 없이" 동작하며, 짧은 수명의 키·인증서를 자동 회전해 준다는 점이다. "비밀번호를 잘 숨기는" 패러다임에서 "비밀번호 자체를 없애는" 패러다임으로의 전환을 원리 수준에서 이해할 수 있다.
