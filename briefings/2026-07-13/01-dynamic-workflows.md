# 오늘의 AI 개념: 다이나믹 워크플로우(Dynamic Workflows)

> 작성일: 2026-07-13 · 분류: claude-code

## 한 줄 정의

Claude Code가 사용자의 큰 작업을 위해 스스로 오케스트레이션 스크립트를 작성하고, 그 스크립트가 백그라운드에서 수십~수백 개의 서브에이전트를 조율해 하나의 결과로 합쳐주는 기능(research preview)이다.

## 쉬운 설명

기존 서브에이전트 방식은 Claude가 대화 turn마다 "다음에 무엇을 시킬지"를 즉흥적으로 결정했다. 다이나믹 워크플로우는 그 결정 로직 자체를 자바스크립트 코드로 미리 짜 놓고, 별도의 실행기(runtime)가 그 코드를 백그라운드에서 그대로 실행하는 방식이다.

비유하면 기존 방식이 팀장이 매 순간 지시를 내리는 즉흥 회의라면, 다이나믹 워크플로우는 팀장이 처음에 전체 진행 순서도(플로우차트)를 문서로 작성해 놓고, 그 문서대로 여러 팀이 동시에 움직이며 중간 결과는 문서 안에만 쌓이는 방식이다.

기존 유사 개념인 스킬(skill)이나 일반 서브에이전트 위임은 매 단계 판단이 Claude의 대화 컨텍스트 안에서 이루어지고 중간 결과도 그 컨텍스트에 쌓인다. 다이나믹 워크플로우는 반복문·분기·중간 결과를 스크립트 변수 안에 가두어 두기 때문에, 사용자 세션의 컨텍스트에는 최종 결과만 남고 실행 자체는 다시 돌려 볼 수 있는 파일로 남는다는 점이 다르다.

## 동작 원리

사용자가 프롬프트에 `ultracode`라는 키워드를 넣거나 "워크플로우를 써서"처럼 직접 요청하면, Claude가 해당 작업에 맞는 자바스크립트 오케스트레이션 스크립트를 작성한다. 스크립트 안에서 `agent()` 함수는 서브에이전트 하나를 생성하고, `pipeline()` 함수는 목록의 각 항목마다 서브에이전트 하나씩을 돌린다.

실행 전에는 계획된 단계 목록과 함께 승인 프롬프트가 뜨며, "실행", "원본 스크립트 보기", "취소" 중 선택한다. 승인 후 런타임은 스크립트를 대화와 분리된 격리 환경에서 실행하며, 동시 실행 에이전트는 최대 16개, 한 번의 실행(run)당 총 에이전트 수는 최대 1,000개로 제한된다.

실행 중에는 사용자가 중간에 개입할 수 없고, 오직 개별 에이전트의 권한 프롬프트만 실행을 잠시 멈출 수 있다. 진행 상황은 `/workflows` 명령으로 언제든 확인하며, 단계(phase)별 에이전트 수·토큰 총량·경과 시간을 볼 수 있고, 일시 중지·재개·특정 에이전트 중단도 가능하다.

## 구체 예시·사례

공식 문서는 예시 프롬프트로 몇 가지 전형을 제시한다. "라우트 핸들러마다 인증 누락을 감사하고 다른 에이전트가 적대적으로 재검증한 뒤 보고하라"처럼 파일 단위 감사·교차 검증형, "`styled-components`를 쓰는 컴포넌트마다 격리된 사본에서 Tailwind로 변환하라"처럼 대규모 병렬 마이그레이션형, "간헐적으로 실패하는 테스트를 두 라운드 연속 새 발견이 없을 때까지 반복 탐색하라"처럼 수렴할 때까지 반복하는 유형이다.

Claude Code에는 이런 워크플로우가 실제로 내장된 사례도 있다. `/deep-research` 명령은 번들로 포함된 워크플로우로, 여러 각도로 웹 검색을 fan-out한 뒤 찾은 출처를 서로 교차 검증하고, 검증에서 살아남지 못한 주장은 걸러낸 인용 보고서를 돌려준다.

## 비슷한 것과 비교

어제 다룬 중첩 서브에이전트(Nested Sub-agents)와는 구조의 축이 다르다. 중첩 서브에이전트는 서브에이전트가 재귀적으로 자기 자신의 하위 서브에이전트를 낳는 "깊이(depth)" 중심 구조로, 최대 5단계까지만 허용되고 각 단계의 위임 여부는 그 단계의 에이전트가 즉흥적으로 판단한다. 다이나믹 워크플로우는 처음부터 Claude가 작성한 스크립트 하나가 "폭(breadth)" 방향으로 수십~수백 개의 에이전트를 병렬 조율하는 구조이며, 판단 로직이 실행 전에 코드로 고정되어 있다는 점이 핵심 차이다.

공식 문서는 서브에이전트·스킬·에이전트 팀·워크플로우 네 가지를 "계획을 누가 쥐고 있는가"로 구분한다. 서브에이전트와 스킬은 Claude가 turn마다 판단하고 결과가 Claude의 컨텍스트에 쌓이며, 에이전트 팀은 리드 에이전트가 공유 작업 목록으로 소수의 장기 실행 동료를 감독한다. 워크플로우만이 유일하게 판단 로직 자체가 스크립트에 있고, 실행 규모도 실행당 수십~수백 개로 다른 셋보다 훨씬 크다.

## 왜 지금 중요한가

다이나믹 워크플로우는 v2.1.150~v2.1.157(2026년 5월 25~29일, Week 22)에 Claude Opus 4.8과 함께 출시됐으며, 공식 What's new 문서는 이를 "research preview" 태그로 표시해 아직 실험적 단계임을 명시한다.

- [Week 22 · May 25–29, 2026 - Claude Code Docs](https://code.claude.com/docs/en/whats-new/2026-w22)

동시 실행 에이전트 16개, 실행당 최대 1,000개라는 구체적 상한이 공식 문서에 명시되어 있어, 한 세션이 감당하기 어려운 규모의 반복 작업(전체 코드베이스 감사, 대규모 마이그레이션)을 이제 세션 하나로 위임할 수 있게 됐다는 점이 의미가 크다.

- [Orchestrate subagents at scale with dynamic workflows - Claude Code Docs](https://code.claude.com/docs/en/workflows)

다만 Pro 요금제에서는 기본적으로 꺼져 있어 `/config`에서 수동으로 켜야 하고, 실행이 25개 이상의 에이전트를 예약하거나 예상 토큰이 150만을 넘으면 "Large workflow" 경고가 뜨는 등, 아직은 비용·범위를 사용자가 직접 통제해야 하는 초기 단계 기능이다.

- [동적 워크플로우로 대규모 서브에이전트 조율하기 - Claude Code Docs(한국어)](https://code.claude.com/docs/ko/workflows)

## 회계법인 AI 직무 연결 포인트

감사에서 표본 검증은 전형적으로 "계정과목 하나당 거래 표본 여러 건"을 반복 처리하는 구조다. 매출채권 잔액 확인처럼 표본 수백 건에 대해 증빙 대사를 수행해야 할 때, 파일(거래 건) 하나당 에이전트 하나를 배정하고 결과를 상위 스크립트가 취합해 예외 건만 사람에게 보고하는 흐름은 공식 예시의 "파일마다 감사 후 교차 검증" 패턴과 그대로 대응된다.

다만 워크플로우 실행 중에는 사용자가 중간에 개입할 수 없다는 제약은 감사 절차의 단계별 승인 관행과 충돌할 수 있다. 파트너·매니저의 단계별 검토가 필요한 절차라면, 문서가 권고하듯 각 단계를 별도의 워크플로우로 쪼개 실행하고 단계 사이마다 사람이 승인하는 설계가 필요하다.

비용 통제 측면에서도 시사점이 있다. 공식 문서는 전체 저장소 대신 디렉터리 하나, 전체 표본 대신 일부 표본으로 먼저 시험 실행해 토큰 지출을 가늠하라고 권고하는데, 이는 감사 실무에서 파일럿 테스트로 절차의 타당성을 먼저 확인한 뒤 전체 모집단으로 확대하는 접근과 원리가 같다.

## 핵심 용어·논쟁

- 오케스트레이션 스크립트 — Claude가 작업 설명을 바탕으로 작성하는 자바스크립트 코드로, 서브에이전트 호출 순서·분기·병렬 실행을 정의한다.
- `agent()` / `pipeline()` — 워크플로우 스크립트 안에서 서브에이전트를 생성하는 함수. `agent()`는 하나, `pipeline()`은 목록의 각 항목마다 하나씩 생성한다.
- ultracode — `/effort ultracode`로 켜는 설정으로, xhigh 추론 강도와 자동 워크플로우 오케스트레이션을 결합해 세션의 모든 실질적 작업에 워크플로우 사용 여부를 Claude가 스스로 판단하게 한다.
- Large workflow 경고 — 실행이 25개 초과 에이전트를 예약하거나 예상 토큰 총량이 150만을 넘을 때 뜨는 권고성 경고로, 실행을 강제로 멈추지는 않는다.

논쟁의 핵심은 "코드로 고정된 판단 로직"의 투명성 문제다. 승인 화면에 "원본 스크립트 보기" 옵션이 있는 것 자체가, 사용자가 수백 개 에이전트를 조율할 스크립트를 실행 전에 실제로 읽고 검토할 필요가 있다는 반증이기도 하다. research preview 태그가 붙어 있고 요금제별로 기본 비활성화되어 있다는 점은, 대규모 자동 실행이 아직 충분히 검증되지 않았다는 Anthropic 측의 신중한 태도를 보여준다.

## 자료 깊이 읽기

### Orchestrate subagents at scale with dynamic workflows — 영어, 공식 문서, 중급
서브에이전트·스킬·에이전트 팀·워크플로우를 "계획을 누가 쥐는가" 기준으로 비교하는 표를 제시하며, 워크플로우만이 판단 로직을 스크립트에 담고 실행당 수십~수백 개 규모로 움직인다고 규정한다. 동시 실행 16개, 실행당 총 1,000개 에이전트 상한, 실행 중 사용자 개입 불가, 워크플로우 자체는 파일시스템·셸에 직접 접근하지 못하고 에이전트를 통해서만 작업한다는 제약을 구체적으로 명시한다. `/deep-research` 같은 번들 워크플로우 실행법, `/workflows`로 진행 상황을 보고 저장하는 법, 저장된 워크플로우에 `args`로 입력을 넘기는 법까지 다룬다.

### 동적 워크플로우로 대규모 서브에이전트 조율하기 — 한국어, 공식 문서, 중급
위 영어 문서의 한국어 번역본으로, 내용이 원문과 동일하게 대응됨을 직접 확인했다. `/config`에서 크기 지침(`small`/`medium`/`large`/`unrestricted`)을 설정해 Claude가 작성할 워크플로우 규모에 조언을 줄 수 있다는 점, 워크플로우 끄기 옵션(설정·환경변수·조직 관리 설정)까지 한국어로 정리되어 있어 한국어 학습자가 세부 설정을 확인하기에 적합하다.

## 자가 점검 질문

1. 다이나믹 워크플로우가 기존 서브에이전트 위임과 다른 점은 "누가, 언제" 다음 단계를 결정하는가로 요약할 수 있다. 이 차이가 실행 결과의 재현성(reproducibility)에 어떤 영향을 주는가?
2. 감사 절차 중 다이나믹 워크플로우로 자동화해도 되는 부분과, "실행 중 사용자 개입 불가"라는 제약 때문에 반드시 단계를 쪼개야 하는 부분을 어떻게 구분할 것인가?
3. research preview 단계의 기능을 회계법인 실무에 도입하기 전에, 어떤 파일럿 검증 절차를 거쳐야 리스크를 통제할 수 있을까?
